## 2. GO/NO-GO判定基準表（5列定義）

**GO/NO-GO判定マトリクス**

| 指標 | GO基準値 | NO-GO基準値 | 中間判定（要確認） | 根拠・信頼度 |
|-----|---------|----------|------------|---------|
| **取引数** | ≥ 2,000 | < 500 | 500～1999 | サンプルサイズ基準（高信頼度） |
| **月利** | ≥ 5% | < 3% | 3～5% | 安定成績目安の下限（中信頼度） |
| **勝率** | 50～70% | ≥ 85% or ≤ 30% | グレーゾーン | リスク型別期待値（中信頼度） |
| **Max DD** | ≤ 18% | > 25% | 18～25% | 制約20%に対する内部基準（高） |
| **Profit Factor** | 1.5～2.0 | ≥ 3.0 or ≤ 1.0 | 1.0～1.5 or 2.0～3.0 | 適正範囲（高信頼度） |
| **最大連敗** | 5～8回 | ≤ 2回 or ≥ 15回 | 3～4回 or 8～15回 | 勝率50%での理論値（中信頼度） |
| **バックテスト期間** | ≥ 12ヶ月 | < 6ヶ月 | 6～12ヶ月 | 相場サイクル基準（中信頼度） |
| **ウォークフォワード乖離** | OOS vs IS ≤ -40% | > -40% | - | パラメータドリフト検出（高） |

**判定ロジック**：複数指標で中間判定が出た場合、デューデリジェンス（ウォークフォワード再検証）が必須。

## 3. 聖杯EA警告＆過剰最適化判定フロー

### 3.1 危険シグナル判定フロー図

```
[バックテスト結果]
      ↓
Step 1：サンプルサイズ確認
├─ 取引数 < 500？
│   ├─ YES → 【NO-GO（即終了）】過剰最適化の可能性極度に高い
│   └─ NO → Step 2 へ
      ↓
Step 2：危険シグナル複合判定
├─ 以下のいずれかに該当？
│   ├─ PF ≥ 3.0 → 利益構造が異常（スプレッド最適化の可能性）
│   ├─ 勝率 ≥ 85% → 逆張り過剰最適化またはナンピン疑い
│   ├─ 最大連敗 ≤ 2 → 環境特異化（特定期間への過度な最適化）
│   └─ YES いずれか → 【NO-GO（即終了）】フォワード崩壊80%超
└─ NO → Step 3 へ
      ↓
Step 3：基本GO条件確認
├─ 月利 ≥ 3% AND 勝率 ≥ 50% AND Max DD ≤ 20%？
│   ├─ YES → Step 4 へ
│   └─ NO → 【要確認】パラメータ見直し
      ↓
Step 4：ウォークフォワード分析実施
├─ IS/OOS乖離 ≤ -40%？
│   ├─ YES → 【GO判定】パラメータドリフト小。リアル稼働候補
│   └─ NO → 【NO-GO】ロジック見直し必須
```

### 3.2 実例：聖杯EA vs 健全なEA の比較

**危険パターン1：最適化過剰EA**

```
バックテスト結果（3ヶ月間）:
- 月利：12%
- 勝率：92%
- PF：4.5
- 取引数：180回
- Max DD：3%

判定：NO-GO
根拠：
  1. 取引数180 < 500 → サンプル不足（統計的信頼性ゼロ）
  2. PF 4.5 ≥ 3.0 → 異常値（スプレッド最適化の可能性）
  3. 勝率92% ≥ 85% → 過度な最適化シグナル
  4. テスト期間3ヶ月 → 相場サイクル未網羅

フォワード予想：フォワード崩壊80%超の高確率
```

**安全パターン：健全なEA**

```
バックテスト結果（12ヶ月間）:
- 月利：5%
- 勝率：55%
- PF：1.8
- 取引数：2,200回
- Max DD：18%
- 最大連敗：6回

判定：GO（ウォークフォワード確認後）
根拠：
  1. 取引数2200 ≥ 2000 → 統計信頼度充分
  2. PF 1.8（1.5～2.0範囲内） → 健全
  3. 勝率55%（50～70%範囲内） → 期待値範囲
  4. Max DD 18% ≤ 20% → 制約クリア
  5. テスト期間12ヶ月 → 相場サイクル網羅
  6. ウォークフォワード乖離-35% ≤ -40% → パラメータドリフト小

リアル稼働適性：高い（月次監視前提）
```

## 4. MQL4/MQL5 既知の罠・バグ予防策

### 4.1 計算精度・インデックスエラーの罠

**罠1：iBarShift・iClose の履歴データ不足**

```
症状：
  バックテスト途中から勝率が0%になる。計算値が NaN または -1 を返す

原因：
  iBarShift(symbol, timeframe, datetime, exact) を使用時、
  過去足のデータが不足していると -1（見つからない）を返す

対策：
  [ ] iBarShift 戻り値を必ずチェック（-1 判定）
  [ ] エラー発生時は OnInit() でスキップフラグを立てる
  [ ] 代わりに「足 shift（現在から N足前）」の固定インデックスを使用推奨
  例：int pos = iBarShift(Symbol(), Period(), TimeCurrent(), false);
      if(pos > 0) { Close[pos] を使用 } else { Skip }
```

**罠2：配列インデックスとバー番号の逆転**

```
症状：
  MT4 では バー [0] = 最新足、バー [1] = 1足前
  MT5 では 逆順（ArraySetAsSeries() で整合）

原因：
  MT4/MT5 混在コーディングで逆順混同

対策：
  [ ] MT4 コードは必ずコメント：// Bar[0]=Latest
  [ ] MT5 では最初に ArraySetAsSeries(Close, false) を実行
  [ ] テスト時に「直近10足の Close[0]～Close[9]」を出力確認
```

**罠3：iOpen/iHigh/iLow/iClose の時刻ずれ**

```
症状：
  バックテストと リアルタイム足の値が異なる（足が確定していない）

原因：
  OnBarOpen イベント中に未確定の足データを参照

対策：
  [ ] OnTick() ではなく、足確定時のタイミングで判定
  [ ] Close[1]（前足確定値）を使用。Close[0]（現在足未確定値）は避ける
  例：if(Bars > 10 && iBarShift(Symbol(), 0, iTime(Symbol(), PERIOD_H1, 0)) == 1)
      { /* 足が新規に確定 */ }
```

### 4.2 ポジション・オーダー管理の罠

**罠4：OrderSelect の戻り値未チェック（致命的バグ）**

```
症状：
  ポジションが正しくクローズされない。不意のダブルクローズ試行。

原因：
  OrderSelect(ticket, SELECT_BY_TICKET, MODE_TRADES) の戻り値をチェック忘れ
  False（失敗）でも処理を続行してしまう

対策：
  [ ] 必ず以下のように書く：
      if(OrderSelect(ticket, SELECT_BY_TICKET, MODE_TRADES)) {
        if(OrderClose(...)) { /* 成功 */ }
        else { /* エラーハンドリング */ }
      } else {
        Print("OrderSelect Failed: ", GetLastError());
      }
  [ ] GetLastError() でエラーコード確認（4xxx = 無効なチケット等）
```

**罠5：同一足での複数エントリー**

```
症状：
  1本の足で複数回エントリーが発火し、同方向ポジション複数作成

原因：
  OnTick() 内でエントリー条件判定 → 同一足の複数 OnTick 呼び出しで条件成立

対策：
  [ ] エントリー条件の外側に「新規足確定」フラグを設定
  [ ] 静的変数 static bool NewBar = false を使用
  例：
      if(iBarShift(Symbol(), 0, iTime(Symbol(), PERIOD_H1, 0)) > 0) {
        NewBar = true;
      } else {
        NewBar = false;
      }
      if(NewBar && (ボリバン+RSI条件)) { OrderSend() }
```

**罠6：MaxDD 計算での未決済ポジション無視**

```
症状：
  MaxDD がバックテストでは小さいが、リアルでは大きくなる

原因：
  バックテスト終了時に未決済ポジションがある場合、
  MaxDD 計算から漏れている（ターミナルログを見ると分かる）

対策：
  [ ] OnDeinit() で全ポジションをクローズする強制処理を入れる
  [ ] バックテスト結果で「未決済ポジション 0個」であることを確認
  [ ] ウォークフォワード分析でも同じチェック
```

### 4.3 リペイント・シグナル遅延の罠

**罠7：前足データ参照による誤シグナル（リペイント）**

```
症状：
  バックテストでは勝率70%だが、フォワード・リアルでは50%に低下

原因：
  指標計算で Close[0]（未確定の現在足）を参照 → 足が確定したら値が変わる（リペイント）
  → バックテストでは「未来情報を知った」状態で最適化されている

対策：
  [ ] Close[1] 以前（確定済み足）のみを指標計算に使用
  [ ] ボリバン・RSI・ATR全て「期間14」→ Close[1]～Close[14] の範囲を参照
  [ ] Close[0]（現在足未確定値）はあくまで「参考値」に留める
  例：
      double bb_upper = iBands(Symbol(), PERIOD_H1, 20, 2, 0, PRICE_CLOSE, MODE_UPPER, 1);
      // 1 = 前足（確定）を参照。0 だと現在足（未確定）を参照 → リペイント
```

**罠8：インジケーターの計算ラグ（遅延）**

```
症状：
  エントリーシグナルが実際のターニングポイントより遅い（既にトレンド反転後）

原因：
  移動平均線・ボリバンは過去14～20足のデータを使用
  → 必然的に価格動きより 7～10足遅れる

対策：
  [ ] 遅延を許容できない戦略の場合：短期パラメータ（7日線等）を使用
  [ ] ただしパラメータを安易に変更すると過度な最適化リスク
  [ ] MVP では「遅延＝安全性」と見なし、標準パラメータを守る
```

## 6. パラメータドリフト検出マニュアル（V1向け）

### 6.1 パラメータドリフトとは

**定義**

```
パラメータドリフト：
バックテスト期間内で最適化されたパラメータが、
新しい相場環境では機能しなくなる現象

例：
- 2024年のトレンド相場で最適化された EMA パラメータが、
  2025年のレンジ相場では勝率30%に低下
```

**ドリフト検出法：MVP検証期間後、V1リアル稼働時の月次パラメータドリフト検出**

```
MVP検証期間：
  バックテスト期間：2025-03-06 ～ 2026-03-05（12ヶ月）
  GO/NO-GO判定：2026-04-30

【V1以降】リアル稼働時の月次ウォークフォワード分析実施：

分割1：
  IS（学習）: 2025-06-06 ～ 2026-02-05（8ヶ月） → 最適パラメータ算出
  OOS（検証）: 2026-02-06 ～ 2026-05-06（3ヶ月） → 新パラメータの成績測定

分割2（ローリング）:
  IS: 2025-08-06 ～ 2026-04-05（8ヶ月） → 再最適化
  OOS: 2026-04-06 ～ 2026-07-06（3ヶ月） → 測定

分割3（同様ローリング）:
  IS: 2025-10-06 ～ 2026-06-05（8ヶ月）
  OOS: 2026-06-06 ～ 2026-09-06（3ヶ月）

ドリフト判定：
  乖離 ≤ -20% → 正常範囲（想定内）
  乖離 -20% ～ -40% → 注意（パラメータ調整検討）
  乖離 > -40% → 警告（ロジック見直し必須）
```

### 6.2 V1以降での月次パラメータドリフト検出手順

**自動検証フレームワーク実装案**

```
V1 実装予定：月次自動ウォークフォワード検証

実行スケジュール（毎月末）:
1. 過去30ヶ月のデータセット準備（リアル稼働後）
2. 8週間単位で分割（3ローリング = 24週間カバー）
3. 各分割で IS/OOS 70:30 テスト実行
4. OOS 成績の平均を計算
5. IS との乖離率を自動計算
6. 乖離 > -40% ならアラート自動生成

検出後の対応フロー：
ドリフト検出
    ↓
【判定】OOS vs IS 乖離率
    ├─ ≤ -20% → 継続稼働（次月監視）
    ├─ -20% ～ -40% → パラメータ微調整検討
    │  └─ 新パラメータで再ウォークフォワード実施
    └─ > -40% → 【停止】ロジック見直し必須
         └─ 根本的な戦略変更 or 新コンセプト検討
```
