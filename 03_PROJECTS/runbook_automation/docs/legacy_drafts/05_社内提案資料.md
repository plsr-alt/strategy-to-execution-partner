# 手順書自動化 — 社内提案資料（10枚構成）
_作成: 2026-02-20 / 提出想定: 社内承認会議_

---

## Slide 1｜背景と課題

**タイトル**: 手作業の手順書入力が、成長の足かせになっている

**箇条書き**
- 現状：手順書（画面キャプチャ含む）を人が目視で読み、システムへ手入力
- 件数増加：業務量の増大に伴い、入力作業時間が月〇〇時間に達している（※実数を入れる）
- ミスの発生：手入力による転記ミスが月〇件発生。対応工数が別途発生
- 属人化：特定担当者のみが処理できる状態。休暇・退職で業務が止まるリスク
- スケール限界：人員増加なしでは、この先の業務量増加に対応できない

**図解案**
```
現状フロー:
[手順書PDF/画像] → [人が目視] → [システムへ手入力] → [確認作業]
     ↑                                                     ↓
  属人化                                              ミス・転記漏れ
```

**トークトラック**
「現在、手順書の入力作業は担当者の目と手に完全に依存しています。件数が増えるほど直線的に時間がかかり、人員を増やしても根本的な解決にはなりません。今日はこれを構造から変える提案です。」

---

## Slide 2｜現状の問題構造

**タイトル**: 問題は3層に積み重なっている

**箇条書き**
- **[効率]** 1件あたり〇〇分の入力作業 → 月〇時間が単純作業に消費
- **[品質]** 手入力ミス率〇% → 後続工程での手戻り・クレーム対応コスト
- **[組織]** 担当者が2名に限定 → バス係数1のリスク
- 根本原因：「画像・PDF情報を人が読んでデジタル化する」プロセスが手動のまま
- 対策なければ：業務量が1.5倍になると入力担当者を1名追加採用が必要（採用コスト○○万円）

**図解案**
```
             品質問題
               ↑
効率問題 ─── [手動入力] ─── 組織問題
               ↓
           スケール限界
```

**トークトラック**
「効率・品質・組織のすべての問題の根本は、画像情報をデジタル化するプロセスが人手のままであることです。ここをAIで自動化することで、3つの問題を一度に解消できます。」

## Slide 3｜To-Be全体像

**タイトル**: 「人が入力する」から「AIが抽出し、人が確認する」へ

**箇条書き**
- 手順書画像を所定フォルダに置くだけで自動処理が開始
- AI（OCR + LLM）がフィールドを抽出し、構造化データに変換
- 信頼度が高い項目は自動でシステムに転記。低い項目のみ人がレビュー
- 全処理ログを自動保存 → 監査・トレーサビリティを確保
- 目標：担当者の作業を「入力」から「確認（1件2〜3分）」に変える

**図解案**
```
To-Beフロー:
[手順書画像] → [AI自動抽出] → [高信頼] → [自動転記]
                    ↓
               [低信頼 30%] → [担当者が確認] → [修正 or 承認]
                                                      ↓
                                               [システム登録]
```

**トークトラック**
「目指す世界は、担当者がゼロから入力する作業をなくすことです。AIが下書きを作り、人は確認だけをする。これにより、判断が必要な業務に時間を使えるようになります。」

## Slide 4｜方式比較

**タイトル**: 実現手段は複数ある。コストと壊れにくさで選ぶ

**箇条書き**
- **OCR単体**: 安いが精度が低く、日本語・複雑レイアウトに弱い
- **RPA（画面操作自動化）**: 設定可能だがUI変更で即壊れる。運用コスト高
- **OCR + ルールエンジン**: 精度は高いが、フォーマット変更のたびにルール修正が必要
- **OCR + LLM（推奨）**: LLMがレイアウト変化・表記揺れを吸収。最も壊れにくい
- **完全LLM（画像直接読み込み）**: 最もシンプルだが、コストとハルシネーション管理が課題

| 方式 | 精度 | 壊れにくさ | コスト | 推奨 |
|------|------|-----------|--------|------|
| OCR単体 | △ | △ | 低 | |
| RPA | ◎ | △ | 高 | |
| OCR+ルール | ○ | △ | 中 | |
| **OCR+LLM** | **◎** | **◎** | **中** | **✅** |

**トークトラック**
「RPAはよく検討されますが、UI変更のたびに壊れます。OCR + LLMの組み合わせは、フォーマット変化をAIが吸収するため、最も運用コストが低い方式です。」

## Slide 5｜推奨案

**タイトル**: AWS Textract + Claude（Amazon Bedrock）+ Google Sheets

**箇条書き**
- **画像読取**: AWS Textract — マネージドOCR。フォーム・表の構造をAPIで取得
- **正規化・JSON化**: Amazon Bedrock（Claude 3.5 Haiku）— 日本語対応LLM。VPC内完結でPII安全
- **出力・レビュー**: Google Sheets — 担当者が使い慣れたUI。承認フロー・共有が容易
- **監査ログ**: S3への全処理ログ自動保存 — 入力画像・AI出力・最終値をすべて記録
- 追加費用概算: 月〇〇件処理で **〜$50〜100/月**（AWSコストのみ）

**図解案**
```
[S3: 画像投入]
     ↓
[Lambda]
     ↓
[Textract]──→ テキスト・表構造
     ↓
[Bedrock Claude]──→ JSON正規化・信頼度スコア
     ↓
[Google Sheets]──→ 担当者レビュー
     ↓
[S3: 全ログ保存]
```

**トークトラック**
「AWSのマネージドサービスを組み合わせるため、サーバー管理は不要です。LLMはAmazon Bedrock経由のため、データが外部に出ません。コストはPoC期間で数千円程度です。」

## Slide 6｜MVP設計

**タイトル**: まず1フローを4週間で動かし、価値を実証する

**箇条書き**
- 対象: 最も件数が多い手順書フロー1種類のみに絞る
- 成果: 4週間後に「画像を置くとSheetsに自動転記される」状態を実現
- 担当者作業: 抽出結果の確認・承認のみ（入力ゼロ）
- 拡張: PoC成功後、フロー数を追加。本番環境への移行はPhase 2以降
- 投資: エンジニア1名 × 4週間 + AWS費用（〜$30）

**図解案（MVPスコープ）**
```
Phase 1（MVP）  Phase 2（本番）  Phase 3（拡張）
    ↓               ↓               ↓
[1フロー]      [3〜5フロー]    [全フロー]
[Sheets出力]   [API登録追加]   [RPA対応]
[4週間]        [〜3ヶ月]       [〜6ヶ月]
```

**トークトラック**
「全フローを一度に自動化しようとすると半年かかります。1フローに絞り、4週間で動くものを作り、価値を実証してから広げます。これが最速のアプローチです。」

## Slide 7｜PoC計画

**タイトル**: 4週間で成功/失敗を明確に判定する

**箇条書き**
- Week 1: AWSインフラ構築・パイプライン疎通確認
- Week 2: 対象フロー1種類のフィールド定義 + 初回精度測定（20件）
- Week 3: プロンプト改善・前処理追加・Sheetsレビューフロー確認
- Week 4: エンドツーエンド50件処理・担当者ウォークスルー・結果評価

**必要なもの（業務側からの提供）**
- テスト用手順書キャプチャ画像 50件以上
- フィールド定義の合意（Week 1中に）
- 担当者によるウォークスルー参加（Week 4）

**トークトラック**
「業務側にお願いするのは、テスト画像50件の提供とフィールド定義の合意、Week 4のウォークスルー参加の3点です。」

## Slide 8｜成功条件

**タイトル**: PoCの合否は数字で判定する

| 指標 | 目標値 | 計測方法 |
|------|--------|---------|
| フィールド抽出精度 | ≥ 85% | 正解セットとの比較 |
| 処理時間 | ≤ 60秒/件 | Lambdaログ |
| レビュー発生率 | ≤ 30% | Sheetsフラグ集計 |
| 手動作業削減率 | ≥ 50% | 従来 vs. PoC後の作業時間比較 |

**箇条書き**
- すべての指標を達成 → Phase 2（本番化）へ進む
- 精度のみ未達 → プロンプト改善を継続（2週間延長）
- 処理時間未達 → アーキテクチャ見直し（Textractの並列化等）
- 改善見込みなし → 別方式（RPA等）の検討に切り替え

**トークトラック**
「PoCは『やってみた』で終わらせません。数字で合否を判定し、Phase 2への投資判断の根拠にします。」

## Slide 9｜リスク

**タイトル**: リスクは3つ。いずれも対策済み

| リスク | 確率 | 影響 | 対策 |
|--------|------|------|------|
| **精度が目標未達（85%未満）** | 中 | 中 | 信頼度スコア + ヒューマンレビューで補完。PoC期間中に改善 |
| **手順書フォーマット変更** | 高 | 低 | LLMがレイアウト変化を吸収。ルールベースより変化に強い |
| **機密情報の外部送信** | 低 | 高 | Amazon Bedrock（VPC内完結）を使用。データが外部LLMに送られない |

**追加考慮事項**
- 担当者がレビューをスキップ → 信頼度低の場合はSheets入力をロック。承認必須にする
- AWS障害時 → S3に画像を保持。障害復旧後に自動リトライ

**トークトラック**
「最大のリスクは精度です。ただし、精度が低い場合でもヒューマンレビューが入るため、誤登録は防げます。PoCはこの精度リスクを定量的に把握するためのものでもあります。」

## Slide 10｜依頼事項

**タイトル**: 3つの意思決定をお願いします

**箇条書き**
1. **PoC実施の承認** — エンジニア1名 × 4週間 + AWS費用（〜3万円）の承認
2. **業務担当者のPoC参加** — テスト画像50件の提供・フィールド定義合意・Week 4ウォークスルー
3. **対象フローの選定** — 「最も件数が多い手順書フロー1種類」を業務側で指定

**スケジュール感**
- 承認: 〇月〇日まで
- PoC開始: 〇月〇日〜
- PoC結果報告: 〇月〇日

**PoC後の判断ポイント**
- 成功 → Phase 2本番化提案（追加費用・スコープ・期間を別途提案）
- 未達 → 原因分析と改善案を提示。Go/No-Goを再判断

**トークトラック**
「今日お願いしたいのは3点だけです。PoCの投資は小さく、リスクも限定的です。4週間後に数字で判断できる状態を作ります。まずPoC実施のGoをいただけますか。」
