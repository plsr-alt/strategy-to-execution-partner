# PoC設計 (PoC Plan)

## 1. 期間
**3週間**（要件定義済みの状態から開始想定）

## 2. 成功条件（定量）
1. 指定したAWSコンソール上の定型作業（例: EC2インスタンスの起動・停止）が、無人で完走すること（成功率 90%以上）。
2. 作業中の画面キャプチャと操作ログが、欠損なく100%取得・保存されること。
3. 取得したデータを元にLLMが生成したMarkdown手順書のうち、人間の手による修正（加筆・訂正）箇所が全体の20%未満であること。
4. UI要素の一部のclass名やIDが変更（モック環境で擬似変更）されても、Self-healing機能でリカバーし操作を完遂できること（成功率 80%以上）。

## 3. 検証項目
- **UI操作の安定性**: 動的なローディング、非同期描画される要素への待機処理が適切に動くか。
- **Self-healingの実効性**: セレクタが見つからなかった際、DOMから代替セレクタをLLMが正しく推定できるか。
- **手順書の品質**: マルチモーダルLLMにより生成された文章が、意図した「操作の意味・理由」を含んでいるか。
- **監査ログの網羅性**: どの要素を、いつ、どんな理由で操作したかが説明可能か。

## 4. 体制・役割
- **プロダクトオーナー/テックリード (1名)**: 全体設計、AI・プロンプトチューニング、評価。
- **自動化エンジニア (1名)**: Playwright実装、ロガー機構の実装。
- **業務担当者 (1名/スポット)**: 自動操作の対象シナリオ（As-Is）の提供、生成された手順書の品質レビュー。

## 5. 成果物
1. PoC用 自動操作＋キャプチャ取得スクリプト（リポジトリ）
2. テスト用モック手順書（生成結果Markdown/PDF）
3. PoC結果報告書（評価結果、課題、コスト/API消費量の実測値）

## 6. Go / No-Go基準
- **Go基準**: 「2. 成功条件」をすべて満たし、1回の生成あたりにかかるLLMトークンコストが許容範囲内（数円〜数十円以内）に収まること。
- **No-Go（中止・ピボット）基準**: AWSなどの複雑なUIにおいて、誤操作がシステムやデータへの破壊的影響を及ぼすリスクが制御しきれない場合。または、LLMによる手順書生成の精度が低く、最初から人間が書いた方が早いと判断された場合。
