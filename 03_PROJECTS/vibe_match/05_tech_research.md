# 顔写真から印象タグを推定するAI — 技術調査

**調査日**: 2026-03-01
**目的**: マッチングアプリ向けに顔写真から「塩顔」「犬系」「きれいめ上品」等の印象タグを推定するシステムの技術選定
**スコープ**: 調査のみ（実装なし）

---

## 1. 使えそうなオープンソースモデル/API

### 1-1. 顔の特徴量抽出

| モデル/ライブラリ | 概要 | ライセンス | 特徴 |
|---|---|---|---|
| **InsightFace** | ArcFace含む顔認識総合ツールキット。検出(RetinaFace) + アライメント + 特徴量抽出 + 属性推定を一括提供 | MIT | 512次元embedding。年齢・性別・表情推定も内蔵。Python/ONNXで軽量推論可 |
| **ArcFace (単体)** | Angular Margin Lossで学習した高精度顔特徴量 | MIT | InsightFaceに内包。単体利用も可。embedding品質が高い |
| **FaceNet** | Googleの顔認識モデル。Triplet Lossで学習 | Apache 2.0 | 128次元embedding。TensorFlow実装多数。やや古いが安定 |
| **CLIP (OpenAI)** | 画像とテキストの共通embedding空間を学習したマルチモーダルモデル | MIT | テキストと画像の類似度計算が可能。Zero-shot分類の王道 |
| **Japanese CLIP (rinna等)** | 日本語テキストに対応したCLIPモデル | 各種 | 「塩顔」等の日本語テキストとの類似度計算に有利 |
| **EVA-CLIP / SigLIP** | CLIPの後継・改良版 | 各種 | 精度向上。HuggingFaceで利用可 |

### 1-2. 顔属性推定

| モデル/ライブラリ | 概要 | 出力属性 |
|---|---|---|
| **DeepFace** | 顔認識・属性推定のPythonラッパー | 年齢、性別、感情、人種（※人種は使用禁止） |
| **InsightFace 属性推定** | InsightFace内蔵の属性推定 | 年齢、性別、3D顔ポーズ |
| **face-api.js** | ブラウザで動くJS顔認識ライブラリ | 68点landmark、年齢、性別、表情 |
| **MediaPipe Face Mesh** | Googleの468点ランドマーク検出 | 高精度ランドマーク（幾何学計算向き） |
| **dlib** | C++/Python顔認識ライブラリ | 68点landmark。形状解析の基盤 |

### 1-3. 「印象」推定に転用可能なモデル

| モデル | 転用方法 |
|---|---|
| **CLIP / Japanese CLIP** | 「塩顔の男性」等のテキストと顔写真のcos類似度を直接計算。Fine-tuning不要でMVPに最適 |
| **FaRL (Face Representation Learning)** | Microsoft Research。顔に特化したCLIP的モデル。顔のセマンティック理解に強い |
| **CelebA属性分類器** | 40属性（魅力的、若い、眼鏡等）の分類器。属性を組み合わせて印象タグにマッピング可能 |
| **Stable Diffusion / BLIP-2 (キャプション生成)** | 顔画像からテキスト記述を生成し、そこから印象ワードを抽出する間接的アプローチ |

## 2. 技術アプローチ候補の評価

### アプローチA: 顔Landmark + 幾何学的特徴 → ルールベース分類

**概要**: 顔のランドマーク（68点 or 468点）から、目の大きさ・鼻の高さ・輪郭形状等の幾何学比率を計算し、ルールで印象タグに変換

**使用技術**: MediaPipe Face Mesh / dlib → 幾何学計算 → if-else / 閾値ルール

**具体例**:
- 目幅/顔幅比 > 0.28 → 「犬系」候補（目が大きい）
- 眉間距離が狭い + 目が切れ長 → 「塩顔」候補
- 顎ラインが丸い + パーツが中央寄り → 「童顔」候補

| 評価軸 | 詳細 |
|---|---|
| **Pros** | 解釈性が高い。GPUほぼ不要（CPU推論可能）。データセット不要。即日プロトタイプ可能。推論コストが最も安い |
| **Cons** | ルール設計が属人的で品質が安定しない。「雰囲気」（髪型・服装・表情の柔らかさ等）は幾何学では捉えられない。タグ追加のたびにルール追加が必要。精度の天井が低い |
| **MVP適合度** | **C（低）** — 「塩顔 vs ソース顔」のような骨格的タグには一定の精度が出るが、「きれいめ上品」「清楚フレッシュ」等の雰囲気タグには対応不可。単独では不十分 |

---

### アプローチB: 顔Embedding → クラスタリング → タグ付け

**概要**: ArcFace等で顔のembeddingを取得し、教師なしクラスタリング（K-means, UMAP+HDBSCAN等）でグループ化。各クラスタに人手でタグを付与

**使用技術**: InsightFace/ArcFace → 512次元embedding → UMAP次元削減 → HDBSCAN

| 評価軸 | 詳細 |
|---|---|
| **Pros** | ラベル付きデータが不要（教師なし）。顔全体の特徴を捉える。新しいクラスタの発見が可能 |
| **Cons** | クラスタが「印象」と一致する保証がない（顔認識embeddingは「同一人物判定」に最適化されており、印象の類似性とは異なる）。クラスタへのタグ付けが人手。再現性・安定性が低い。クラスタ数の決定が困難 |
| **MVP適合度** | **D（不適）** — 顔認識用embeddingは「誰か」の識別に最適化されており「どんな雰囲気か」の軸と直交する可能性が高い。MVP段階での労力対効果が悪い |

---

### アプローチC: CLIP/マルチモーダルモデルで「印象テキスト」との類似度

**概要**: CLIPモデルに顔画像を入力し、各印象タグのテキスト（「塩顔の男性」「犬系の優しそうな顔」等）とのcos類似度を計算。上位タグを割り当て

**使用技術**: OpenAI CLIP (ViT-L/14) or Japanese CLIP (rinna/japanese-clip-vit-b-16) → cos similarity

**具体的なフロー**:
1. 顔検出 + クロップ（InsightFace RetinaFace）
2. CLIP画像エンコーダで画像embedding取得
3. 事前定義したタグテキスト群をCLIPテキストエンコーダでembedding取得（キャッシュ可）
4. cos類似度を計算し、上位N個のタグを返す

**プロンプトエンジニアリングの工夫**:
- 単純な「塩顔」ではなく「a photo of a man with a salt face, sharp eyes, thin features」のようにプロンプトを拡張
- 日本語CLIPなら「あっさりした塩顔の男性の写真」等
- 複数プロンプトのアンサンブルで精度向上

| 評価軸 | 詳細 |
|---|---|
| **Pros** | **ラベル付きデータ不要（Zero-shot）**。テキストを変えるだけでタグ追加・変更が即座に可能。「雰囲気」を含むセマンティックな類似度を計算可能。実装が最もシンプル（推論パイプラインのみ） |
| **Cons** | CLIPは顔に特化していないため、背景・服装・照明にも影響を受ける（→顔クロップで緩和可能）。日本語固有の概念（塩顔等）の理解度に限界がある可能性。精度のチューニングがプロンプト依存 |
| **MVP適合度** | **A（最適）** — Zero-shotで即日MVP構築可能。プロンプト調整だけで品質改善ができ、反復が速い。後からFine-tuningへの移行も容易 |

---

### アプローチD: Fine-tuned分類器（教師あり学習）

**概要**: 印象タグのラベル付き顔画像データセットを構築し、分類器をFine-tuning

**使用技術**:
- ベースモデル: CLIP / ViT / EfficientNet
- 学習: マルチラベル分類（1枚に複数タグ）
- データ: クラウドソーシングでアノテーション

| 評価軸 | 詳細 |
|---|---|
| **Pros** | 精度が最も高くなる可能性。タスク特化により「塩顔」等の独自概念を正確に学習可能。運用フェーズでの推論が高速 |
| **Cons** | **ラベル付きデータが必要（最低タグあたり500~1000枚推奨）**。アノテーションコストが高い（印象は主観的→アノテータ間一致率が低い）。データ収集の肖像権問題。初期構築に2~4週間。タグ追加のたびに再学習 |
| **MVP適合度** | **B（中）** — 精度は最高だが初期コストが高い。MVP後のv2で検討すべき。CLIPのZero-shotで収集したデータを使ってFine-tuningする段階的アプローチが合理的 |

---

### アプローチ比較まとめ

| | A: ルールベース | B: クラスタリング | C: CLIP Zero-shot | D: Fine-tuned |
|---|---|---|---|---|
| **初期構築速度** | 1~2日 | 3~5日 | **1~2日** | 2~4週間 |
| **必要データ** | なし | 数百枚(ラベル不要) | **なし** | タグあたり500~1000枚 |
| **「雰囲気」対応** | 不可 | 低 | **高** | 最高 |
| **タグ追加容易性** | ルール追記 | 再クラスタリング | **テキスト追加のみ** | 再学習 |
| **精度天井** | 低 | 低~中 | 中~高 | **最高** |
| **推論コスト** | **最安** | 中 | 中 | 中~安 |
| **MVP推奨度** | C | D | **A** | B |

## 3. MVP推奨アプローチ

### 推奨: アプローチC（CLIP Zero-shot）+ アプローチAの部分併用

#### 構成

```
[入力: 顔写真]
    ↓
[1. 顔検出 + クロップ] ← InsightFace RetinaFace
    ↓
[2. 基本属性取得(補助)] ← InsightFace (性別判定のみ使用 → タグセット切替)
    ↓
[3. CLIP画像Embedding] ← Japanese CLIP or OpenAI CLIP ViT-L/14
    ↓
[4. 印象タグテキストとのcos類似度計算]
    ↓
[5. スコア上位タグを返却 + 信頼度付き]
    ↓
[出力: {tags: ["塩顔", "クール", "知的"], scores: [0.82, 0.76, 0.71]}]
```

#### 印象タグの設計例

**男性向け**:
- 顔系統: 塩顔 / ソース顔 / 醤油顔 / 砂糖顔
- 動物系: 犬系 / 猫系 / うさぎ系 / オオカミ系
- 雰囲気: クール / 知的 / 爽やか / ワイルド / 中性的

**女性向け**:
- 雰囲気: きれいめ上品 / 清楚フレッシュ / クール / キュート / ナチュラル
- 動物系: 犬系 / 猫系 / うさぎ系 / 狐系
- 系統: フェミニン / ボーイッシュ / 大人っぽい / 童顔

#### CLIPプロンプト設計戦略

各タグに対して複数のプロンプトを用意し、平均スコアで判定:

```
"塩顔" → [
  "a Japanese man with a salt face, thin sharp features, light impression",
  "あっさりした薄い顔立ちの日本人男性",
  "a man with delicate thin facial features and cool expression"
]
```

**ポイント**: 英語CLIPと日本語CLIPの両方でスコアを取り、加重平均する手法が有効。日本語固有概念は日本語CLIPの重みを上げる。

#### 補助的なルールベース（アプローチA併用）

MediaPipe Face Meshから以下の幾何学特徴を補助シグナルとして使用:
- 目の縦横比（切れ長 vs 丸目）
- 顔の縦横比（面長 vs 丸顔）
- 眉と目の距離（彫りの深さ指標）

これをCLIPスコアに加算する形で精度補強（重み0.2程度）。

#### 必要なデータ量とコスト

| 項目 | 内容 |
|---|---|
| **学習データ** | 不要（Zero-shot） |
| **検証データ** | 100~200枚（人手でタグ付けしてCLIPの精度検証用） |
| **プロンプト調整** | 1~3日の試行錯誤 |
| **初期開発工数** | 3~5人日 |
| **クラウドコスト（検証）** | ほぼゼロ（CPU推論可能。GPU使用でもColab無料枠で十分） |

## 4. プライバシー/倫理の技術的ガードレール

### 4-1. 禁止推定の防止

| リスク | 技術的対策 |
|---|---|
| **人種推定** | DeepFaceの`race`属性は一切使用しない。モデル出力から人種関連の次元を削除/マスク。CLIPプロンプトに人種関連語を含めない |
| **年齢断定** | InsightFaceの年齢推定値は「年代」としてのみ使用（20代/30代等）。具体的な年齢数値はユーザーに返さない |
| **性別バイナリ強制** | 性別判定はタグセット切替の内部処理のみに使用。ユーザーの自認性別を優先する設計にする |
| **容姿のランク付け** | 「美人度」「イケメン度」等のスコアリングは設計から除外。タグは序列ではなく「分類」のみ |

**実装方針**:
- 出力フィルタ層を設け、禁止ワードリスト（人種名、具体的年齢、容姿評価語）に該当する出力をブロック
- CLIPの印象タグ候補リストをホワイトリスト方式で管理（事前承認されたタグのみ出力可能）

### 4-2. 有名人類似出力の防止

| 対策 | 詳細 |
|---|---|
| **有名人embeddingデータベースを持たない** | そもそも有名人の顔データを保持しないことで、類似度計算自体を不可能にする |
| **CLIPプロンプトに人名を含めない** | タグ候補に「○○似」を一切入れない |
| **出力テンプレート固定** | 自由テキスト生成を行わず、事前定義タグの選択のみに限定することで、意図しない有名人言及を防止 |

### 4-3. データ取り扱い

| 項目 | 方針 |
|---|---|
| **顔画像の保存** | 推論後即時削除。サーバーに顔画像を保持しない（ステートレス推論） |
| **Embedding保存** | 顔embeddingも保存しない（embedding→顔再構成の研究が進んでいるため） |
| **ログ** | 入力画像・embedding・推論結果を紐づけるログを残さない |
| **オプトアウト** | ユーザーが過去のタグ結果を削除できるAPIを用意 |

## 6. MVP推奨構成まとめ

### 技術スタック

| レイヤー | 選定 | 理由 |
|---|---|---|
| **顔検出** | InsightFace RetinaFace (ONNX) | 軽量・高精度・CPU推論可 |
| **印象推定（主）** | Japanese CLIP (rinna/japanese-clip-vit-b-16) + OpenAI CLIP ViT-L/14 | Zero-shotで即構築。日本語概念対応 |
| **印象推定（補助）** | MediaPipe Face Mesh → 幾何学特徴 | 骨格系タグの精度補強 |
| **推論基盤** | Google Cloud Run (CPU, 2vCPU) | コスト最安。オートスケール |
| **API** | FastAPI + ONNX Runtime | 軽量・高速 |
| **フロント** | 既存アプリからAPIコール | REST API |

### 推定精度の目安

- **骨格系タグ（塩顔/ソース顔等）**: 70~80%（CLIP + 幾何学補助で比較的高精度）
- **雰囲気系タグ（きれいめ上品/清楚等）**: 55~70%（CLIPの得意領域だが主観的で一致率に天井あり）
- **動物系タグ（犬系/猫系等）**: 60~75%（CLIPがメタファー理解をある程度持つ）

※「印象」は主観的なため、人間同士の一致率が60~70%程度。モデルがこの範囲に入れば十分実用的。

### 開発ロードマップ

| フェーズ | 期間 | 内容 |
|---|---|---|
| **Phase 1: MVP** | 1週間 | CLIP Zero-shot + 顔検出パイプライン構築。プロンプト調整。検証データ100枚で精度確認 |
| **Phase 2: 精度改善** | 2~3週間 | プロンプト最適化。幾何学特徴の補助シグナル追加。検証データ拡充（500枚） |
| **Phase 3: Fine-tuning** | 1~2ヶ月 | Phase 1-2で蓄積したユーザーフィードバックを使い、CLIP or ViTをFine-tuning。精度大幅向上 |
| **Phase 4: 本番最適化** | 随時 | ONNX量子化。バッチ推論。キャッシュ最適化。レイテンシ改善 |

### 最初の1歩

1. InsightFace + Japanese CLIP をローカルで動かす
2. 印象タグ20種のプロンプトを設計
3. 手元の顔画像10枚で動作確認
4. 「なんとなく合ってる」レベルを確認できたらCloud Runにデプロイ

## 補足: 代替アプローチの備忘

- **GPT-4V / Claude Vision等のLLMに顔画像を投げる**: 精度は高いが、1回$0.01~0.03と高コスト。レイテンシも2~5秒。プライバシーポリシー上の課題（第三者APIに顔送信）。MVPの検証用として部分的に使う価値はある（正解ラベル作成のオラクルとして）
- **Stable Diffusion系のface encoder**: 顔の審美的特徴を捉えている可能性があるが、研究段階
- **FaRL (Microsoft)**: 顔特化のCLIP的モデル。精度が高い可能性があるが、日本語対応なし。Phase 2以降で検証候補
