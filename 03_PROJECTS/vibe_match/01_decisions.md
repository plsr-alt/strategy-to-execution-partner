## Q1. タグ体系の確定（大カテゴリ/サブタグの最終リスト）

### 推奨案: 3軸 × 8-10大カテゴリ（性別別）

**3つの軸:**
1. **系統軸**（調味料系）: 塩顔 / ソース顔 / 醤油顔 / 砂糖顔
2. **キャラクター軸**（動物系）: 犬系 / 猫系 / うさぎ系 / オオカミ系（男性）/ 狐系（女性）
3. **雰囲気軸**（スタイル系）: クール / ナチュラル / きれいめ / フレッシュ / 知的

**出力形式:** 各軸から1つずつ、合計Top5タグ（軸横断で確率順）

### 根拠
- 調味料系・動物系は日本で広く認知されている（SNSバズ実績あり、炎上リスク低）
- 3軸にすることで「補完マッチ」の相性行列が設計しやすい
- CLIP Zero-shotのプロンプト設計と相性が良い（各タグの言語表現が明確）

### 炎上リスク対策
- **NG表現**: 美人/イケメン/ブス/不細工/点数/ランク/偏差値
- **注意表現**: 童顔（→「フレッシュ」に言い換え）、老け顔（→使わない）
- 全タグをポジティブな印象語に統一（ネガティブ分類を作らない）

### 決定事項
- [ ] タグ最終リストは `06_tag_taxonomy.md` で確定（調査エージェント結果を反映後）
- [ ] 炎上リスクレビューの外部監修（スタイリスト等）は入れるか？

## Q2. ターゲットの恋活/婚活比率

### 推奨案: **恋活メイン（7:3）で開始**

### 根拠
- **市場調査より**: 恋活層（20-30代前半）がマッチングアプリのメインユーザー
- **「雰囲気マッチ」のコンセプト**: 「並んだ時に映える」は恋活の方がフィットする
- **UX設計の簡素化**: 婚活は条件フィルタ（年収、学歴等）が重要 → MVP複雑化を避ける
- **競合**: with/Pairs が恋活50%:婚活50%。タップル/Tinder が恋活80%。差別化するなら「恋活×雰囲気」の掛け合わせが空白

### 具体的な影響
| 項目 | 恋活メイン | 婚活メイン |
|------|----------|----------|
| UI | カード探索重視、カジュアル | 条件フィルタ重視、詳細プロフィール |
| 課金 | ブースト・スーパーいいね（スポット） | プレミアム月額（高単価） |
| ターゲット年齢 | 22-32歳 | 28-38歳 |
| トーン | 軽やか・楽しい | 真剣・安心感 |

### 代替案
- 婚活メイン → 単価は高いが市場は小さい。Dine/東カレと競合
- 両方均等 → UXが中途半端になるリスク

## Q3. 画像保存方針

### 推奨案: **サーバ推論 + 短期保存（30日） + 暗号化**

### 根拠
- MVP段階で端末推論はモデルサイズ制約（CLIP ViT-Lは300MB+）で非現実的
- 都度推論（保存しない）は再診断のたびにアップロードが必要でUX悪い
- 30日保存なら「写真追加→再診断」のフローがスムーズ

### セキュリティ対策
| 対策 | 内容 |
|------|------|
| 暗号化 | AES-256 (AWS KMS) |
| アクセス | プリサインURL（15分有効） |
| 削除 | ユーザー操作で即時削除 + 30日TTL自動削除 |
| CDN | CloudFront署名付きURL |
| ログ | 画像URLをマスキング |

### 代替案
- **保存しない（都度推論）**: プライバシー最強だがUX×。再診断のたびに再アップロード必要
- **端末推論**: 将来のCOULD。モバイル向けCLIPの軽量化が進めば検討

## Q4. 年齢確認の厳格度

### 推奨案: **MVPは簡易（18歳以上チェック + SMS認証）→ βでeKYC導入**

### 根拠
- **出会い系サイト規制法**: インターネット異性紹介事業に該当する場合、年齢確認が義務
  - ただし「マッチング後のチャット」が異性紹介に該当するかは解釈による
  - **MVPは安全側に倒して年齢確認を実施すべき**
- SMS認証で電話番号を確保 → 多重アカウント抑止にも有効
- eKYCはCVR低下リスクが高い（身分証撮影で離脱率30-50%の事例あり）→ βで導入して影響測定

### 段階的導入
| フェーズ | 方式 | CVRへの影響 |
|---------|------|-----------|
| MVP | 18歳以上チェックボックス + SMS認証 | 低 |
| β | 身分証の生年月日確認（AI-OCR） | 中（離脱率+15-30%想定） |
| 正式 | 本人確認（eKYC） + 高信頼バッジ | 高（任意にすべき） |

## Q5. 相性行列（補完定義）の作り方

### 推奨案: **初期は手動設計（ファッション/スタイリング知見ベース） → データ駆動で更新**

### 根拠
- 完全データ駆動はMVP段階でデータ不足（マッチ実績が必要）
- ファッション・スタイリングの「対比の法則」は確立されている
- 初期行列があれば、ユーザーの「いいね/スキップ」データで重みを学習更新できる

### 初期相性行列（例）

| 統一感（近い） | 補完（映える） |
|--------------|--------------|
| 塩顔 ↔ 塩顔 | 塩顔 ↔ ソース顔 |
| 犬系 ↔ 犬系 | 犬系 ↔ 猫系 |
| ナチュラル ↔ ナチュラル | ナチュラル ↔ クール |
| きれいめ ↔ きれいめ | きれいめ ↔ ワイルド |
| フレッシュ ↔ フレッシュ | フレッシュ ↔ 大人クール |

**設計原則:**
- 統一感 = 同じ軸の同じor隣接タグ
- 補完 = 同じ軸の対極タグ（柔↔シャープ、明↔暗、親しみ↔クール）
- 相性行列は `06_tag_taxonomy.md` で詳細定義

### 更新計画
1. MVP: 手動行列（上記）
2. β: マッチ率・会話率が高いペアのタグ組み合わせを集計
3. 正式: 協調フィルタリングで行列を自動更新

## Q6. 収益化の初期パッケージ

### 推奨案: **無制限いいね + ブースト（スポット）の2点**

### 根拠
- **市場調査**: 全アプリ共通の課金導線は「いいね制限解除」と「露出アップ」
- **MVPの課金設計原則**: シンプルに始めて、何に金を払うか検証する
- 既読・詳細フィルタは「あると嬉しい」だがMVPでは過剰

### 価格案
| 項目 | 価格 | 根拠 |
|------|------|------|
| プレミアム月額（無制限いいね + 優先表示） | ¥3,600/月 | with/Pairs帯に合わせる |
| ブースト（1回 / 30分露出UP） | ¥600/回 | Tinder/Bumble帯 |
| 女性 | 基本無料 | 市場標準。女性ユーザー確保が最優先 |

### βで追加検討
- 既読（チャット内）
- 詳細フィルタ（タグ指定検索）
- スーパーいいね（理由メッセージ付き）

## Q7. 運用体制（モデレーション）

### 推奨案: **MVP = 自動 + オーナー1人の人手レビュー（平日のみ）**

### 根拠
- MVP段階はユーザー数が少ない（~1000人想定）→ 通報数も限定的
- 自動化で8割カバー（NGワード、スパムパターン）
- 残りは日次で人手確認（1日15分想定）

### 運用設計
| レイヤー | 方式 | 内容 |
|---------|------|------|
| L1: 自動 | NGワードフィルタ | チャット/プロフィールの暴言・性的表現を自動ブロック |
| L1: 自動 | スパム検知 | 同一メッセージ連投、URL含むメッセージを自動フラグ |
| L2: 人手 | 日次レビュー | 通報キュー確認（平日1回、15分想定） |
| L3: エスカレーション | オーナー判断 | アカウント停止、法的対応の判断 |

### βでの強化案
- 通報SLA: 24時間以内に一次対応
- 休日対応: 外部モデレーション委託（CrowdWorks等）
- AIモデレーション: チャット内容のトーン分析
