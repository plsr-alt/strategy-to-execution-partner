# Agent Factory — プロジェクト CLAUDE.md

## 概要
公式ドキュメントを唯一の事実源として、AIエージェントシステムを自動生成する"工場"。

## フォルダ構造
```
agent_factory/
├── sources/        ← Layer 0: 唯一の事実源（公式ドキュメント等）
├── digest/         ← Layer 1→2 中間層: 圧縮された知識（数百行）
├── system/         ← Layer 3: 生成されたエージェントシステム出力
├── pipeline/       ← パイプライン定義・テンプレート
├── CLAUDE.md       ← 本ファイル（PJルール）
├── PLAN.md
├── SPEC.md
├── TODO.md
└── KNOWLEDGE.md
```

## レイヤー構造
| Layer | 内容 | 格納先 |
|-------|------|--------|
| 0 | 公式ドキュメント（原文） | `sources/` |
| 1 | 設計ガイド（矛盾併記・タグ付き） | `digest/guide_*.md` |
| 2 | 圧縮知識（数百行digest） | `digest/compressed_*.md` |
| 3 | ドメイン特化エージェントシステム | `system/<domain>/` |

## タグ規約（必須）

### 信頼度タグ
| タグ | 意味 | 使い方 |
|------|------|--------|
| `⚠️公式未記載` | sourcesに根拠がない情報 | 推測・経験則で書いた箇所に付与 |
| `⚠️検証未通過` | fact-checkerが検証できなかった | 修正→再検証で上限回数超えた箇所 |
| `⚠️矛盾あり` | 複数ソース間で矛盾 | **両方の記述を併記**し、判断材料を残す |
| `✅検証済` | fact-checkerが検証通過 | sourcesとの照合が完了した箇所 |

### 運用ルール
- **書いてないことは原則書かない**。書くなら必ず `⚠️公式未記載` タグ
- **矛盾は消さない**。両方併記して `⚠️矛盾あり` タグ。判断はユーザーが行う
- タグなし = sourcesに根拠あり＋検証済み、と見なせる状態にする

## パイプライン（二人体制）

### 原則: writerとcheckerは必ず分離する
同じAI/同じコンテキストで「書く」と「疑う」を兼任させない。

### ステップ
```
[Phase 1: 取得]
  fetch → sources/ に保存

[Phase 2: 執筆＋検証ループ]
  section-writer（執筆）→ fact-checker（検証）
    ├─ PASS → 次セクションへ
    └─ FAIL → 修正 → 再検証（上限3回）
              └─ 上限超え → ⚠️検証未通過 タグで残す

[Phase 3: 統合＋圧縮]
  全セクション統合 → 設計ガイド完成
  設計ガイド → digest（数百行に圧縮）

[Phase 4: 生成]
  digest + ドメイン要件 → ジェネレーター → system/ に出力
```

### WSL+Groq 委譲ルール
- **Phase 1（取得）**: Claude Code がURL一覧を決定 → WSL+Groq or スクリプトで取得
- **Phase 2（執筆＋検証）**: WSL+Groq に委譲（長文生成のため）
- **Phase 3（統合＋圧縮）**: WSL+Groq に委譲
- **Phase 4（生成）**: WSL+Groq に委譲、Claude Code がレビュー

## 生成プロセス（工場の型）

### Phase A: ヒアリング（7ラウンド）
1. ドメイン・目的の確認
2. ユーザー像・ペルソナ
3. 主要ワークフロー
4. エッジケース・制約
5. 既存システムとの連携
6. 品質基準・KPI
7. 漏れ/矛盾の検証（別エージェント）

### Phase B: 設計
- system-architect がエージェント構成を設計
- reviewer が設計を検証

### Phase C: 構築
- agent-builder がエージェント/スキル/ルールを生成

### Phase D: ガイド生成
- guide-writer が利用ガイド/クイックスタートを生成
- 出力は `system/<domain>/` に格納
