# マーケティングチーム設計スキル

ブランド専用のマーケティングチーム設計書を、6ワーカー分業制で直接生成する。
CEO/部長のルーティングオーバーヘッドを省略し、確立済みのパイプラインを直接実行する。

## 起動方法

```
/marketing-team 恋愛改善設計士
/marketing-team 03_PROJECTS/love_improvement_designer
/marketing-team <ブランド名>
```

**自動トリガー**: 以下の条件で自動起動を提案する（詳細は末尾「自動フロー統合」参照）
- 新プロジェクトフォルダに brand_kit や business_sheet が揃った時
- 「おはよう」で `12_marketing_team_design.md` が未作成のプロジェクトを検出した時

---

## Phase 0: 情報収集（自動・確認不要で進む）

### 0-1. プロジェクトフォルダ特定
`$ARGUMENTS` を解析:
- `03_PROJECTS/xxx` 形式 → そのまま使用
- ブランド名のみ → `03_PROJECTS/` 配下を Glob で探索
- 見つからない → ユーザーに確認

### 0-2. 既存資料の自動探索
`PROJECT_DIR` 配下 + `02_STRATEGY/` + 関連プロジェクトから自動発見:

```
Glob: {PROJECT_DIR}/**/*.md
Glob: 02_STRATEGY/**/*.md
```

| パターン | 用途 |
|---------|------|
| `*brand_kit*`, `*brand*` | ブランド定義 |
| `*business*`, `*sheet*` | ビジネスモデル |
| `*kpi*`, `*validation*` | KPI・撤退基準 |
| `*x_post*`, `*投稿*` | 既存コンテンツ |
| `*schedule*`, `*launch*` | スケジュール |
| `PLAN.md`, `SPEC.md` | 4ドキュメント方式 |

### 0-3. 制約条件の自動抽出
発見したファイルから以下を抽出。8割以上読み取れたら確認不要で Phase 1 へ:

- 運営体制 / 可処分時間 / 月予算 / 顔出し可否
- 得意な形式 / 避けたい運用 / 使用メディア / 目的

不足分のみ AskUserQuestion で確認。

---

## Phase 1: 6ワーカー直接オーケストレーション

CEO/部長を経由せず、スキル自身が直接6ワーカーを管理する。
パターンは確立済み（strategy部門 / heavy / 6名体制）。

### タスクディレクトリ作成
```
TASK_DIR = company/tasks/{YYYY-MM-DD}_marketing_{ブランド名スラッグ}/
mkdir: {TASK_DIR}/20_worker_outputs/
```

### Wave 1: 調査＆抽出（並列実行）

Worker 1 と 2 は依存関係なし。Agent ツールで同時に background 起動する。

**Worker 1: worker-researcher**（バックグラウンド）
```
タスク: {ブランド名}の市場調査・競合分析・参考事例・ターゲットインサイト調査
指示:
  (1) X上の競合マップ（カテゴリ別に2-3名ずつ）
  (2) 該当ジャンルのnote市場動向・購買傾向
  (3) 構造的アプローチで成功している参考人物・企業（具体名必須）
  (4) ターゲット層のインサイト（悩み・行動パターン・購買心理）
  (5) 制約条件（顔出し不可等）での成功事例
  Web検索を活用。推測には[推測]タグ。
出力: {TASK_DIR}/20_worker_outputs/01_research.md
```

**Worker 2: worker-extractor**（バックグラウンド）
```
タスク: 背景資料から情報を全抽出・整理
入力: Phase 0-2 で発見した全ファイル
指示:
  (A) ブランド定義（タグライン・バイオ・差別化・NGトーン）
  (B) ターゲットペルソナ（心理・行動・痛点・購買動機）
  (C) ファネル設計（各ステップの役割と離脱リスク）
  (D) KPI・撤退基準・改善レバー
  (E) 既存コンテンツ資産
  (F) 制約条件の影響範囲
  (G) ツールアセットの活用可能性
  (H) 将来展開
出力: {TASK_DIR}/20_worker_outputs/02_extract.md
```

→ 両方の完了を待つ

### Wave 2: 構造化（順次実行）

**Worker 3: worker-structurer**
```
タスク: 7セクション骨格設計・必須要件11項目マッピング
入力: 01_research.md + 02_extract.md
指示:
  - 必須要件11項目 → セクション対応表
  - 組織図: 本人/AI/ツール 3層、10-15ロール
  - 各担当詳細: 7項目統一フォーマット
  - メディア別: 文脈/インサイト/訴求/NG表現/形式 の5軸
  - コンテンツ変換フロー: テキスト図
  - 30日・90日プラン: Phase分け・マイルストーン
  - レビュー体制: 週次・月次チェックポイント
出力: {TASK_DIR}/20_worker_outputs/03_structure.md
```

### Wave 3: ドラフト

**Worker 4: worker-drafter**
```
タスク: 7セクション完全ドラフト生成（3000字超）
入力: 03_structure.md + 02_extract.md
指示:
  - 03_structure.md の骨格に厳密に従う
  - 必須要件11項目を全カバー
  - 制約条件内で実行可能な設計
  - 参考事例は 01_research.md から具体名引用
  - 週次業務量が可処分時間内に収まること
出力: {TASK_DIR}/20_worker_outputs/04_draft.md
```

### Wave 4: 品質チェック

**Worker 5: worker-critic**
```
タスク: 批評・欠落チェック・制約適合性・ブランドポリシー検証
入力: 04_draft.md
指示: 8観点で検証
  (1) 完全性: 7セクション + 11項目カバレッジ
  (2) 実行可能性: 週次業務量が制約内か
  (3) 予算適合性: 月予算制約に反する施策がないか
  (4) ブランドポリシー: NGトーンの混入がないか
  (5) KPI整合性: 既存KPIと整合しているか
  (6) 全否定部隊の機能性: 独立性が担保されているか
  (7) 矛盾・重複: 役割間の責任重複がないか
  (8) 改善提案: editor向けの具体的改善指示
出力: {TASK_DIR}/20_worker_outputs/05_critic.md
```

### Wave 5: 最終編集

**Worker 6: worker-editor**
```
タスク: 批評反映・最終編集・完成版生成
入力: 04_draft.md + 05_critic.md
指示:
  - critic指摘を全て反映
  - 冒頭にエグゼクティブサマリー（3段落）
  - 末尾に「改善ポイント・次フェーズへの課題」セクション追加
  - 文体統一（ですます調・読みやすく）
  - Markdownフォーマット整備
出力（2箇所に保存）:
  - {TASK_DIR}/20_worker_outputs/06_final.md
  - {PROJECT_DIR}/12_marketing_team_design.md
```

## Phase 2: 軽量レビュー＆報告

CEO/部長の代わりに、スキル自身が最終チェックを行う:

1. `06_final.md` を Read して以下を確認:
   - 7セクション + エグゼクティブサマリー + 改善ポイント = 9ブロック存在するか
   - 必須要件11項目のキーワードが含まれているか（簡易grep）
   - 制約条件に反する記述がないか

2. ユーザーに報告:
   - 成果物の場所: `{PROJECT_DIR}/12_marketing_team_design.md`
   - 70点の成果物: 主要セクションの要点（箇条書き）
   - 残り30点: 改善ポイント・次アクション

## 出力セクション構成

| # | セクション | 内容 |
|---|----------|------|
| 0 | エグゼクティブサマリー | 全体像・勝ち筋・使い方 |
| 1 | 全体設計 | 思想・勝ち筋・避けるべき戦い方 |
| 2 | 組織図 | 役割一覧（本人/AI/ツール3層）・依存関係・兼務 |
| 3 | 各担当の詳細 | 役職名・目的・責任・KPI・連携・週次業務・成果物 |
| 4 | メディア別戦略 | 文脈・インサイト・訴求・NG表現・最適形式 |
| 5 | コンテンツ変換フロー | 元コンテンツ→各媒体の変換ルール |
| 6 | 実行計画 | 30日プラン（週単位）・90日プラン（月単位） |
| 7 | レビュー体制 | 全否定部隊・週次/月次レビュー・改善ループ |
| 8 | 改善ポイント | 残課題・次フェーズ方向性 |

## 自動フロー統合

このスキルは手動起動だけでなく、以下のタイミングで自動的に起動提案される。

### トリガー1: 「おはよう」ルーティン
DAILY.md 読み込み時に、`03_PROJECTS/` 配下のアクティブプロジェクトをスキャンし、
`12_marketing_team_design.md` が存在しないプロジェクトがあれば提案:

```
「{プロジェクト名} にマーケティングチーム設計書がまだありません。
 /marketing-team {プロジェクト名} で作成しますか？」
```

### トリガー2: プロジェクト初期セットアップ完了時
4ドキュメント方式（PLAN.md → SPEC.md → TODO.md）の SPEC.md が完成した段階で:

```
「SPEC.md が完成しました。次のステップとしてマーケティングチーム設計を
 自動生成できます。実行しますか？」
```

### トリガー3: 「マーケ」「SNS戦略」等のキーワード検出
ユーザーの発言に以下のキーワードが含まれる場合、このスキルを自動提案:
- マーケティング / マーケ / SNS戦略 / メディア戦略 / コンテンツ戦略
- ファネル設計 / マーケプラン / 集客設計

## /work との違い

| 項目 | `/work` 経由 | `/marketing-team` 直接 |
|------|------------|---------------------|
| CEOルーティング | あり（1エージェント消費） | なし（パターン確定済み） |
| 部長WBS分解 | あり（1エージェント消費） | なし（WBS固定テンプレ） |
| ワーカー数 | CEOが判定 | 固定6名 |
| 並列実行 | 部長が判定 | Worker 1&2 を必ず並列 |
| 品質レビュー | CEO再レビュー（1エージェント） | スキル内で軽量チェック |
| **合計エージェント数** | **9-10回** | **6回** |
| **節約** | — | **約30-40%のトークン節約** |
