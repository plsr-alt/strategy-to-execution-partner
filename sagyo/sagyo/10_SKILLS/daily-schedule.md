# Skill: daily-schedule

## 目的

プリセールスエンジニアの1日を「成果物の完成」中心に設計する。完璧主義を排し、ver1を最速で仕上げることに最適化した行動計画を生成する。

## 設計原則

1. **成果物単位に分解**: 「資料作成」ではなく「提案書p1-5のver1完成」のように書く
2. **午前＝最重量の成果物1つ**: 最も認知負荷の高い資料を午前に集中して仕上げる
3. **同時進行は最大2つ**: 時間帯ごとにアクティブなタスクは2つまで
4. **修正依頼は午後まとめ**: 午前に受けたフィードバックは午後の修正ブロックで一括処理
5. **ver1完成が最優先**: 80点のver1を出す > 100点を目指して未完成。MITの完了基準は「ver1が共有できる状態」

## トリガー

- 「おはよう」
- 「今日の予定は？」

## 入力

ユーザーからの追加指示（任意）:
- 今日の固定予定（商談・デモ・社内会議など）
- 締切が迫っている案件
- 体調・コンディション
- 特別な制約（短縮勤務・移動時間など）

## 参照ファイル/データ

| 優先順 | ソース | 取得内容 |
|--------|--------|---------|
| 1 | Google Calendar（連携時） | 商談・デモ・会議の時刻 |
| 2 | GitHub Projects（連携時） | 未完了Issue・期限付きタスク |
| 3 | `06_OPERATIONS/schedules/` 直近の日報 | 前日の申し送り・未完了タスク |
| 4 | `03_PROJECTS/*.md` | 進行中案件のタスク・期限・成果物一覧 |
| 5 | `07_FINANCE/KPI.md` | 当月KPI進捗と乖離 |
| 6 | `00_INBOX/*.md` | 未整理の受信タスク・依頼 |

## 手順

### Step 1: 情報収集
1. 上記参照ソースを順に読み込む
2. 外部連携不可の場合はローカルファイルのみで進行

### Step 2: 成果物の洗い出しと分解
- 抽象タスクを**具体的な成果物**に変換する:

| NG（抽象） | OK（成果物単位） |
|-----------|----------------|
| 提案書を作る | 提案書p1-5構成案のver1完成 |
| デモ準備 | デモシナリオ3本のスクリプトver1完成 |
| 技術検証 | 検証結果レポート(結論+根拠3点)のver1完成 |
| 見積もり対応 | 見積書(構成案+概算)のver1完成 |

- 成果物ごとに**完了基準を1文で定義**する（＝これが終わればver1として共有できる状態）

### Step 3: 分類
**緊急度 × 重要度**
| | 重要 | 非重要 |
|------|------|--------|
| **緊急** | A: 即実行 | B: 短時間で片付ける |
| **非緊急** | C: 時間を確保して実行 | D: 削除/延期 |

**認知負荷（プリセールス基準）**
- **高**: 提案書の新規構成、技術設計、RFP回答 → 午前に配置
- **中**: 既存資料の改修、見積作成、調査 → 午後前半に配置
- **低**: メール返信、修正依頼対応、社内連絡 → 午後後半/隙間に配置

### Step 4: MIT選定
- **MIT-1**（午前の主役）: 認知負荷が最も高い成果物を1つ選ぶ。午前いっぱいで ver1完成を目指す
- **MIT-2, MIT-3**: 午後に完了させる成果物。MIT-1より軽いもの
- 完了基準: 「ver1として関係者に共有できる状態」＝完了

### Step 5: 工程表作成

**タイムブロック構造（プリセールス最適化）**

| 時間帯 | 用途 | ルール |
|--------|------|--------|
| 09:00-09:15 | 朝の計画確認 | スケジュールレビュー、当日の商談確認 |
| 09:15-12:00 | **午前集中ブロック** | MIT-1に全振り。割り込み対応しない |
| 12:00-13:00 | 昼休憩 | — |
| 13:00-14:30 | 午後ブロック1 | MIT-2または商談・デモ |
| 14:30-14:45 | 休憩 | — |
| 14:45-16:00 | **修正まとめブロック** | 午前〜午後に溜まった修正依頼を一括処理 |
| 16:00-16:45 | 午後ブロック2 | MIT-3または残タスク消化 |
| 16:45-17:00 | 日報・翌日仕込み | 申し送り作成、明日の成果物を仮リスト化 |

- 15分刻みで配置
- 商談・デモは固定予定として先に埋め、残り時間にタスクを割り当て
- 90分連続作業ごとに15分休憩を挟む
- **同一時間帯のアクティブタスクは最大2つ**

### Step 6: リスク先読み
- ボトルネック（顧客回答待ち・技術検証の未確定要素）を洗い出す
- 時間超過リスクのあるタスクにバッファ案を付記
- **完璧主義アラート**: 見積もり2h超の成果物は「スコープを削れないか」を問う

## 出力

```markdown
# Daily Schedule: YYYY-MM-DD

## MIT（今日の成果物）
1. [ ] {成果物名} ver1完成 — 完了基準: {1文}
2. [ ] {成果物名} ver1完成 — 完了基準: {1文}
3. [ ] {成果物名} ver1完成 — 完了基準: {1文}

## 成果物分解
| 成果物 | 緊急×重要 | 認知負荷 | 見積時間 | 完了基準 |
|--------|----------|---------|---------|---------|
|        | A/B/C/D  | 高/中/低 |         |         |

## 工程表
| 時間 | 内容 | 区分 | 同時進行 |
|------|------|------|---------|
| 09:00-09:15 | 朝の確認 | 定型 | — |
| 09:15-12:00 | {MIT-1} | 午前集中 | 単独 |
| ... | ... | ... | ... |
| 14:45-16:00 | 修正依頼まとめ対応 | 修正ブロック | — |

## リスク・詰まりの先読み
- ⚠️ {リスク} → 対策: {案}

## 完璧主義チェック
- {成果物名}: ver1で十分か？ 削れるスコープはないか？

## 前日からの申し送り
- {内容}
```

## 保存先

```
06_OPERATIONS/schedules/YYYY-MM-DD_schedule.md
```

- 日付はスケジュール対象日
- 1日1ファイル。同日に再生成した場合は上書き更新

## 出力例

```markdown
# Daily Schedule: 2026-02-21

## MIT（今日の成果物）
1. [ ] X社向け提案書(p1-10) ver1完成 — 完了基準: 構成・訴求ポイント・概算費用が入った状態で営業に共有できる
2. [ ] Y社デモシナリオ ver1完成 — 完了基準: 3ユースケースのスクリプトが通しで読める
3. [ ] Z社RFP回答(技術要件部分) ver1完成 — 完了基準: 全設問に1文以上の回答が埋まっている

## 成果物分解
| 成果物 | 緊急×重要 | 認知負荷 | 見積時間 | 完了基準 |
|--------|----------|---------|---------|---------|
| X社提案書 p1-10 ver1 | A | 高 | 2.5h | 営業に共有できる状態 |
| Y社デモシナリオ ver1 | A | 中 | 1.5h | 通しで読めるスクリプト |
| Z社RFP回答(技術) ver1 | B | 中 | 1h | 全設問に回答が埋まっている |
| 社内技術ブログ下書き | C | 中 | 1h | (延期→明日) |
| A社からの修正依頼3件 | B | 低 | 45min | 修正ブロックで対応 |
| メール・Slack返信 | B | 低 | 20min | 隙間で処理 |

## 工程表
| 時間 | 内容 | 区分 | 同時進行 |
|------|------|------|---------|
| 09:00-09:15 | 朝の確認・商談スケジュール確認 | 定型 | — |
| 09:15-11:45 | X社向け提案書 p1-10 ver1作成 | 午前集中/MIT-1 | 単独 |
| 11:45-12:00 | メール・Slack返信 | 隙間処理 | — |
| 12:00-13:00 | 昼休憩 | — | — |
| 13:00-14:30 | Y社デモシナリオ ver1作成 | 午後ブロック1/MIT-2 | 単独 |
| 14:30-14:45 | 休憩 | — | — |
| 14:45-15:30 | A社修正依頼3件まとめ対応 | 修正ブロック | 最大2件並行 |
| 15:30-16:30 | Z社RFP回答(技術要件) ver1 | 午後ブロック2/MIT-3 | 単独 |
| 16:30-16:45 | バッファ / 突発対応 | — | — |
| 16:45-17:00 | 日報・明日の成果物リスト仮作成 | 定型 | — |

## リスク・詰まりの先読み
- ⚠️ X社提案書: 導入事例の数値を事例担当に確認中 → 対策: 仮数値で埋めて注釈、確認取れ次第差替
- ⚠️ Y社デモ: ステージング環境のデータが古い → 対策: 午前中にデータ更新を依頼、間に合わなければスクショで代用

## 完璧主義チェック
- X社提案書: デザイン調整はver2以降。テキストと構成が伝わればver1完了
- Z社RFP: 回答が1文でも「埋まっている」ならver1。詳細化はレビュー後

## 前日からの申し送り
- X社: 営業の佐藤さんから訴求ポイント3点を受領済。反映して構成に入る
- Y社: デモ日程は来週水曜で確定
- A社修正: Slackで3件受領。いずれも軽微（文言修正+図差替+数値更新）
```
