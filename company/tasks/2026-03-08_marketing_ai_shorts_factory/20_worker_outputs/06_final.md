## エグゼクティブサマリー

### 事業の本質

AI Shorts Factory は、既存 YouTube パイプライン資産（moviepy / Groq / EC2環境）を再活用しながら、FLUX.2[klein] + Ken Burns + 直接API組み合わせで、**完全ゼロコスト（月額 ¥100のGCSのみ）** の自動ショート動画生成エンジンです。毎日3本の短編動画（15-60秒）を生成し、YouTube Shorts / Instagram Reels / TikTok の3プラットフォームに同時展開することで、既存チャネル登録者の10倍リーチを目指します。差別化の核は「金融×AI」「教育×AI」など **複合テーマのニッチ化** + **週次最適化AI** による品質維持です。

### 市場機会

AI生成ショート動画市場は2023-2026年にかけてCAGR 45-60%で成長中。特に日本国内では「長尺動画を見る時間がない層」の AI学習需要が高く、「金融×AI図解」「トレンド解説」といったテーマで検索ニッチを獲得できます。YouTube AdSense + TikTok Creator Fund + BOOTH デジタル販売の3本柱で、**月0本投稿時 ¥0 → 90本投稿時 月¥5-15万の推定月収** を実現する構図が成立します。

### 実行上の課題と対策

制約条件は3つ：**予算ゼロ**（無料ツール化）、**1人運用**（完全自動化必須）、**X API有料化除外**（YouTube/IG/Pinterest限定）。対策は確立済みです。Phase 0（セットアップ）→ Phase 1（パイプライン構築）→ Phase 2（テスト投稿・GO/NO-GO判定）→ Phase 3（本運用）の4フェーズで段階的に実行し、Phase 2終了時に AVG_VIEWS ≥ 500 / エラー率 < 5% / 規約違反 0件 の3条件で本格展開判定を行います。

## セクション1：全体設計 — AIショート動画自動量産の思想・勝ち筋

### 1.1 コンセプト定義

**AI Shorts Factory** は、「AIが毎日描く、3プラットフォーム向けショート動画の自動工場」です。

| 項目 | 内容 |
|------|------|
| **タイトル** | AI Shorts Factory — 無料AIアートで毎日の発見を |
| **本質** | FLUX.2[klein] + Ken Burns + Groq による完全ゼロコスト自動生成エンジン |
| **出力物** | YouTube Shorts / Instagram Reels / TikTok 向けショート動画（15-60秒）毎日3本 |
| **既存資産再活用** | youtube_automation.py の moviepy / Groq / EC2 環境を転用 |

### 1.2 事業目的（4本柱）

1. **既存 YouTube パイプライン資産の活用** — 開発済みのスクリプト・環境を Shorts 生成へ転用し、開発コスト 0 を実現
2. **リーチ最大化** — YouTube Shorts 発見度 + Instagram 保存度 + TikTok エンゲージメントの加重レバレッジで、月ビュー 3,000 → 300,000 への成長シナリオを構築
3. **収入源多角化** — YouTube AdSense（月¥30-50K）+ Instagram Bonus（月¥5-10K）+ TikTok Creator Fund（月¥10-15K）で月計 ¥45-75K の収益基盤を構築
4. **Rena キャラ育成への接続** — ショートフォーマットでの認知度構築 → キャラ販売・デジタルグッズへの流入チャネル創設

### 1.3 差別化ポイント（5つの勝ち筋）

| ポイント | 実装内容 | 根拠 |
|---------|---------|------|
| **コスト構造** | ¥0/月実現（EC2共用+GCS¥100のみ）。Midjourney/Runway など有料モデルを完全除外 | 既存リソース活用。無料OSS化で参入障壁なし |
| **テンプレ多様化** | 単一テンプレ駆動を脱却。4テンプレ → 7テンプレへ段階拡張（金融/教育/トレンド/AI制作/ビフォーアフター等） | 競合は「AI画像ギャラリー」に特化。複合テーマで差別化 |
| **テーマニッチ化** | 「金融×AI」「教育×AI」など複合テーマで検索ニッチ獲得。単一テーマのみライバルを避ける | 市場調査で「金融情報飢餓」確認。テーマ別AVG_VIEWS で検証 |
| **高頻度投稿信号** | 3本/日で「アクティブ」判定 → アルゴリズム有利（競合は平均 1-2本/日） | YouTube/TikTok API ドキュメント。高頻度が推奨度向上 |
| **動画品質** | Ken Burns アニメーション + Wan 2.1 フォールバック で「静止画→動画」品質感を維持 | FLUX画像の静止性を補う。VBench 86% の Wan 2.1 活用 |

### 1.6 インフラ前提条件 — EC2リソース共用の検証基準

**既存 youtube_automation.py との共用可能性**

本設計書では既存 YouTube パイプライン資産（EC2 t4g.medium）とAI Shorts Factory を同一インスタンスで運用する前提で ¥0予算を実現しています。以下の検証を Phase 0 で完了する必要があります：

| 検証項目 | 内容 | 判定基準 |
|---------|------|--------|
| **実行時間帯の競合確認** | youtube_automation 実行時間帯と 06:00-07:30 の重複確認 | 重複なし、または±1時間シフト可能 |
| **VRAM メモリ共用可能性** | ComfyUI + moviepy 同時メモリ使用量が VRAM 8GB 以下か実測 | ピーク使用量 ≤ 7.5GB で安全余裕確保 |
| **GPU スケジューリング** | CUDA/Metal 両プロセス間の逐次実行可能性を確認 | リソース競合なし / タイムアウト0件 |
| **GCS ストレージ容量** | 3本/日 × 30日 = 90本 × ~200MB/本 = ~18GB/月が ¥100で賄えるか | 標準ストレージ ¥1.71/GB で ¥30.78/月 + 30日自動削除 = ¥100内 |

**共用不可判定時の代替案**:
- EC2 インスタンス追加（月 ¥3,000-5,000 追加コスト）
- または AI Shorts Factory 実行時間を 18:00-19:00 へシフト（パイプライン遅延＋手動QA時間帯変更）

### 3.1 AI層（4つのシステム）

#### ロール3: 台本・プロンプト生成エンジン（Groq LLM）

| 項目 | 内容 |
|------|------|
| **システム名** | Groq API（mixtral-8x7b）|
| **目的** | 毎日のテーマ × 動画台本 × プロンプト を自動生成 |

| **アウトプット** | JSON形式：台本 / プロンプト / BGMテンポ / キャプション（フォーマット統一） |
| **制約** | 禁止キーワード（人物/顔/実在人物）自動フィルタリング |
| **実行時間** | 90秒/本（3本並列） |
| **エラーハンドリ** | 生成失敗時は テンプレ台本フォールバック → リトライ3回 |

#### ロール4: 画像生成エンジン（FLUX.2[klein]）

| 項目 | 内容 |
|------|------|
| **システム名** | FLUX.2[klein]（4B, Apache 2.0） |
| **目的** | Groq 台本 → AI画像 3-5枚 を高速生成 |
| **環境** | ComfyUI on EC2 t4g.medium（VRAM 8GB） |
| **実行時間** | 0.5秒/枚 × 4枚 = 2秒/本（含シード固定） |
| **品質チェック** | 人物顔検出 → 自動削除 / 暴力的表現フィルタ |
| **フォールバック** | VRAM不足時 → Wan 2.1 / AnimateDiff（段階的） |
| **1日処理量** | 3本 × 4枚 = 12枚/日 |

#### ロール5: 動画化・アニメーションエンジン（Ken Burns + Wan 2.1）

| 項目 | 内容 |
|------|------|
| **システム名** | FFmpeg Ken Burns + Wan 2.1（フォールバック） |
| **目的** | 静止画 → 短尺動画（15-60秒）への変換 |
| **処理フロー** | FLUX画像（4枚） → Ken Burns ズーム/パン → Wan 2.1（オプション）で動画化 |
| **出力形式** | MP4（H.264, 1080p, 30fps） |
| **実行時間** | Ken Burns 15秒/本（含エンコード） |
| **フォールバック階層** | Ken Burns（15秒） → Wan 2.1（30-60秒 [推測]）→ AnimateDiff（90-120秒 [推測]）。CogVideoX は使用禁止推奨 |
| **注記** | Wan 2.1 以降のフォールバック時に実行時間が2-8倍増加するため、毎日06:00-07:30 実行枠の逸脱リスクあり。Phase 1 中にComfyUI環境で各モデルの実行時間を実測し更新必須 |
| **1日処理量** | 3本/日 |

#### ロール6: 編集・合成エンジン（moviepy）

| 項目 | 内容 |
|------|------|
| **システム名** | moviepy + Groq 字幕生成 |
| **目的** | 動画 + BGM + 字幕 + テキストOL を最終合成 |
| **処理内容** | 1. BGM挿入（Pixabay/YT Audio）/ 2. 字幕同期（Groq→SRT）/ 3. テキストOL（テーマ/プロンプト） |
| **出力形式** | MP4（H.264, 1920x1080, 30fps） |
| **実行時間** | 30秒/本（含BGM同期） |
| **QA** | 音声ズレチェック（±200ms許容）/ テキスト重複検出（3行以上NG） |
| **1日処理量** | 3本/日 |
