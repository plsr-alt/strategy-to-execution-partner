## 良い点

- **セクション5の パイプラインフロー（タイミング図）が秀逸**: 毎日06:00-07:30 の自動処理と手動QA を時系列で明確に描出。各ステップの実行時間（4.5分/台本生成、2秒/画像、45秒/動画化等）が具体的で再現性が高い
- **セクション7の 3段階品質ゲート設計**: 自動チェック → ルールベースチェック → 本人QA承認 の段階的フィルタリングが実装可能で、品質担保の仕組みが堅い
- **付録A のチェックシート**: 投稿前に何をチェックするか一覧化。署名欄もあり、実行者責任が明確
- **撤退基準が明記されている**（セクション8.3）: 「即撤退トリガー」「要評価」「撤退時アクション」の3段階で、中止時の学習資産保存・ドキュメント化が考慮されている
- **メディア別戦略（セクション4）の詳細**: YouTube/Instagram/TikTok の3プラットフォーム別アルゴリズム特性（「発見」「保存」「シェア」）と投稿タイミング・CTA戦略が差別化されており、運用現場の指針として機能する
- **版管理・次回更新日を明記**: v1.0、作成日2026-03-08、次回更新2026-04-08（Phase 2終了）と継続管理の態勢が示されている

## 改善必須事項（リリース前に対応必要）

### 1. **週次業務量の正確な記述**（セクション2.2）

**現状**:
```
週次業務: 月1回 GO/NO-GO 判定会議・KPI確認・外部リスク監視（30分）
```

**問題**: 毎日の投稿前チェック（3-5分/本 × 3本 = 9-15分/日）と週次規約確認（1時間）が週次業務欄に入っていない。セクション5.1「月計9.5時間/月」と矛盾。

**修正案**:
```
| 役割2: QA・品質責任者 | **週次業務** | 日次：投稿前チェック（9-15分/日） / 週次：規約アップデート確認（1時間） / 月次：KPI・撤退基準確認（30分） |

→ 月計: 9.5時間 = 週2.4時間（手動QA） + 土曜手動投稿補充時間0-1時間 = 週2.4-3.4時間
  * 理想値「毎日30分以内」の達成は「投稿前チェック3分/本×3本=9分 + 自動処理10分」で計19分/日。手動補充なしの場合のみ30分内に収まる可能性あり
```

**影響**: セクション2 全体で「実際の運用負荷」を透明化。CLAUDE.md の「毎日30分以内が理想」を達成・達成不可どちらかを明確化

### 2. **Phase 2 GO判定の統計的有意性を補強**（セクション6.3）

**現状**:
```
GO条件（全クリア必須）:
- AVG_VIEWS ≥ 500（7日平均）✅
```

**問題**:
- 「7日平均」と書かれているが、P2-1（3/25-3/31）は7日間 → 約21本投稿で統計が成立
- P2-4（4/1-4/7）も同様
- 合計28本のサンプルで AVG 500 は妥当か？標本サイズ根拠なし
- テーマ別パフォーマンス（付録B で金融×AI 420、AIアート 280、教育×AI 150）がテスト時と本運用時で安定するか未検証

**修正案**:
```
GO判定のデータ要件（新規追加）:
- サンプルサイズ最小 28本以上（P2全期間で達成見込み）
- 各テーマ別に最小 3-4本 × テーマ数で統計的有意性確保
- データ不足時（例：ある1テーマが1本のみ投稿）→ Phase 3 開始予定通りだが、そのテーマは β版として段階投稿
- テーマ別の信頼区間（±100ビュー）を計算し、AVG 500 達成の蓋然性を評価
```

**追記箇所**: セクション6.3 の P2-GO/NO-GO判定 直前に「データ検証ガイドライン」セクションを新規追加

### 3. **全否定部隊の権限・エスカレーションパス明記**（セクション7.3）

**現状**:
```
全否定部隊の権限: 提案段階でのリスク指摘 → CEO報告 / Phase 2 NO-GO 可能性の事前指摘 → 改善方策立案 / 撤退判断の最後のセーフティネット
```

**問題**:
- 「提案段階」の意味が曖昧（誰に提案するのか）
- Phase 2 中に全否定部隊が指摘を出した場合、修正するか修正を拒否するか、その判定者は誰か
- GO/NO-GO判定時に全否定部隊の指摘が無視された場合の異議申し立てメカニズムが無い

**修正案**:
```
全否定部隊の権限（改訂版）:

【タイミング】
- 毎週日曜 KPI 分析時に全否定部隊が「テーマ妥当性」「技術リスク」等を検査
- Phase 2 終了時（4/8）の GO/NO-GO 判定 前に最終チェック実施

【判定フロー】
  全否定部隊 指摘作成
    ↓
  CEO（本人）へ報告
    ↓
  CEO 判定: 「修正対応」/ 「指摘受け入れて NO-GO」/ 「指摘は参考だが GO 進行」
    ↓
  修正対応の場合 → 次週までに改善方策を実装 → 全否定部隊が再検査

【修正拒否権】
  CEO が「指摘は参考だが GO 進行」を決定した場合、全否定部隊は議事録にその旨を記載
  → Phase 3 以降、リスク顕在化時の原因追跡可能化
  （全否定部隊に最終拒否権はなし。最後の意思決定は本人 CEO）

【成果物】
  - 毎週の全否定部隊チェックレポート（テンプレ作成）
  - Phase GO判定時の「全否定部隊が検査完了した」署名欄を追加
```

**追記箇所**: セクション7.3 の末尾に「権限範囲・エスカレーション図」セクションを新規追加

### 4. **EC2 リソース共用の可能性を検証記述へ**（セクション1.2 / 1.4）

**現状**:
```
コスト構造: ¥0/月実現（EC2共用+GCS¥100のみ）
```

**問題**: 既存 youtube_automation.py（YouTube 動画化）と AI Shorts Factory（ショート動画自動生成）が同じ EC2 t4g.medium 上で動作する場合、
- VRAM 競合（moviepy + FLUX.2[klein] の同時実行可能か）
- CPU スケジューリング（毎日06:00-07:30 のショート生成と youtube_automation の実行時間帯の競合）
- ストレージ（GCS ¥100で本当に30日のバックアップが賄えるか）

**修正案**:
```
制約条件の根拠付与（新規セクション 1.6「インフラ前提条件」）:

【既存 youtube_automation.py との共用可能性】
  - 既存スクリプト実行時間帯: 記載なし（CLAUDE.md 参照 or 直接確認必要）
  - 提案時間帯: 毎日 06:00-07:30 の集中実行（±1時間の時間帯ずれ可能）
  - 検証項目:
    ✓ youtube_automation 実行時間が 06:00-07:30 と被らないか確認
    ✓ ComfyUI + moviepy の同時メモリ使用量が VRAM 8GB 以下か実測
    ✓ GPU スケジューリング（CUDA/Metal）が両プロセス間で逐次実行できるか確認
  - 共用不可判定時: EC2 インスタンス追加か、別時間帯実行へシフト

【GCS ¥100/月の容量確認】
  - 3本/日 × 30日 = 90本 × ~200MB/本 = ~18GB/月
  - GCS 標準ストレージ ¥1.71/GB → 18GB = ¥30.78/月
  - 30日自動削除で容量リサイクルする前提で ¥100/月 内で賄えるか確認
  - または古いファイルは自動削除（7日以上経過）に変更
```

**追記箇所**: セクション1.2 に新規 「1.6 インフラ前提条件」セクションを追加。出口条件として「既存スクリプトとの共用可能性検証完了」を Phase 0-7 のチェックリストに追加
