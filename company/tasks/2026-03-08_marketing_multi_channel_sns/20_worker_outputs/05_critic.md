## 総合評価

**B (72点/100点)**

◎完全性：90% ◎実行可能性：75% ◎予算適合性：100% ◎ブランドポリシー：80% ◎KPI整合性：70% ◎全否定部隊機能性：60% ◎矛盾・重複：80% ◎改善提案具体性：85%

**評価コメント**
ドラフトは全体構成・視点設定共に優れており、9セクション + 改善ポイント + 要約が完全に揃っている。が、以下3項目で**リリース前の修正必須**。

1. KPI目標値の矛盾（Phase 1 vs Phase 2 vs Phase 3）
2. 全否定部隊の独立性・エスカレーション経路が明記されていない
3. X手動投稿の「毎日30分以内」という制約がSection 3 で定義されていない

これらの修正により B → A（85点以上）への昇格が見込める。

## 良い点

- ✅ **全9セクション完全構成** — エグゼクティブサマリー、全体設計、組織図、詳細役割、メディア別戦略、実行計画（30日/90日）、レビュー体制、リスク検証、改善ポイント。全必須要件を網羅
- ✅ **実行性が高い** — Phase 0/1/2/3 の4段階ロードマップが明確。マイルストーン・完了条件・責任者が各タスクで定義されている
- ✅ **予算制約を厳密に守着** — EC2共用、各API無料枠内で設計。有料ツール（Ayrshare等）の混入ゼロ
- ✅ **1人運用を前提に設計** — X手動投稿以外のすべてを自動化。不要な人的介入を排除
- ✅ **プラットフォーム別の訴求最適化が具体的** — Instagram/Pinterest/X の特性（リーチ寿命、ユーザー属性、形式）を正確に記載。「テンプレート例」で実装イメージを明確化
- ✅ **リスク認識が深い** — 「Groq月200呼び出し上限」「X API有料化」「プロンプト品質低下」「YouTube品質低下」の4大リスクを先制的に列挙。対応策・代替案も用意
- ✅ **KPI・撤退基準が定量的** — 「投稿数 < 20本」「流入 < 月200回」など、判断を曖昧にしない

## 改善必須事項（リリース前に対応必要）

### 高優先度（修正必須）

**1. KPI目標値の矛盾を解消**
- **現状**: Phase 1「月500-1000訪問」→ Phase 2「月1000-1500訪問」→ Phase 3「月2000+訪問」
  - 各Phase の「新規フロー」と「累積フロー」の区別がない
  - 達成困難な段階的上昇に見える
- **修正内容**:
  ```
  Phase 1終了（4月5日）:
  - 累積訪問流入: 月500-1000
  - 登録者増: 月+50

  Phase 2終了（4月30日）:
  - 累積訪問流入: 月1000-1500（Phase 1比+30%）
    理由: A/Bテスト勝者採用 + 投稿時間最適化
  - 登録者増: 月+100（Phase 1比+50%）

  Phase 3（5月以降）:
  - 累積訪問流入: 月2000+（YouTube増産 + プラットフォーム拡張による）
  - 登録者増: 月+200+ 予測
  ```
- **修正箇所**: Section 6.2 L468-473、L489-492、L509-514

**2. 全否定部隊の独立性・権限を明記**
- **現状**: 「リスク TOP 5」「撤退判断基準」を列挙しているだけ
  - Devil's Advocate が実際にどの権限を持つのか不明
  - CEO 判断に対する拒否権なし
- **修正内容**: Section 8 に「8.3 全否定部隊の独立性」を新規追加
  ```
  ### 8.3 全否定部隊の独立性・権限

  #### 役割定義
  - **名称**: Devil's Advocate / 全否定監視委員
  - **メンバー**: Claude（分析AI）+ CEO 判断のピアレビュー
  - **独立性**: CEO 判断に対する異議申し立て権を持つ（下記参照）

  #### 権限
  1. **延期要求権**: CEO が Phase 移行判定した場合、
     リスク評価が「未対応」の項目があれば延期要求可能
  2. **追加テスト要求権**: プロンプト品質低下を検知した場合、
     A/Bテスト追加実施の要求可能
  3. **月次報告権**: 月末レビューで独立した「リスク評価レポート」を提出
     （CEO提出の「月次KPI達成度」とは別）

  #### エスカレーション経路
  - リスク「🔴高」で対応策が実装未の場合 → CEO と協議
  - 3項目以上のリスク「🔴高」が未対応 → Phase 移行延期
  ```

**3. X手動投稿の週次業務負荷を定量化**
- **現状**: 「毎朝6-7時：テキスト確認・修正・投稿」と定性的に記載
  - 「30分以内」という制約がどこにも見当たらない
  - 実際の運用可能性の根拠が薄い
- **修正内容**: Section 3「Role (2)」に以下を追加
  ```
  | 項目 | 内容 |
  |------|------|
  | **週次業務時間** | **毎日30分以内**（詳細: 朝6-7時=20分、昼監視=5分、金曜報告=5分） |
  | **朝6-7時業務** | ①Groq生成テキスト受信（自動）②品質確認（テキスト150字以下？等）③修正必要なら2分以内に実施④X投稿（コピペ+投稿ボタン1分以内） |
  | **昼監視** | エラーログ Slack 確認（5分）。投稿成功 or エラー報告 |
  | **金曜報告** | 週次エラー報告書作成（テンプレート形式、5分） |
  | **超過時対応** | 20分超過した場合は CEO に即報告。週次協議で対応 |
  ```
- **修正箇所**: Section 3 L162-173、Section 7.3 に「X投稿完了時間」を追加

### 中優先度（推奨修正）

**4. Instagram 3時差投稿のアルゴリズムペナルティ検証**
- **追加**: Section 4.4「制約対応」に以下を追加
  ```
  | 制約 | 検証状況 | 対応策 |
  |------|---------|--------|
  | **同一テキスト3回投稿によるペナルティリスク** | 未検証 | Phase 1 Week 3-4 で A/B テスト実施：<br/> (a) 同一テキスト3回投稿（統制群） <br/> (b) テキスト微修正版3パターン投稿（実験群）<br/> 両群の ER 差 > 20% なら (b) に統一 |
  ```

**5. Groq プロンプト検証テストを Phase 0 に組み込む**
- **追加**: Section 6.1「Phase 0」タスク欄に
  ```
  | 日時 | マイルストーン | タスク追加 |
  |------|-------------|---------|
  | **Week 1** | **API申請・環境準備** | ... ⑤ Groq プロンプト検証テスト（最小10パターンの YouTube メタで 3PF 用テキスト自動生成→仕様合致度判定） |
  ```

**6. NG 表現チェックリストを整備**
- **追加**: Section 9.1 に新規項目
  ```
  ### NG表現チェックリスト（自動+目視確認）

  #### 自動スキャン項目
  - 「絶対に」「確実に」「必ず」→ 表現修正
  - 金融規制対象キーワード（「投資」「運用」「利回り」など）→ 「家計管理」に統一
  - 医療機器関連表現 → スキップ
  - 過度な誇大表現（「~だけで」「~するだけ」が連続） → 修正

  #### 運用者による最終目視確認
  - ① X投稿前に必ず確認（高リスク）
  - ② Instagram / Pinterest は Groq 自動生成の場合抽出検査（10投稿に1回）
  ```
