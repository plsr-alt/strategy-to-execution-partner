# EA企画書：コンセプト企画～MVP検証計画

## 1. 企画概要

本企画は、プロのクオンツPM兼MT4/MT5エンジニアとして「検証可能で実装可能で運用可能」なEA（自動売買）戦略を設計するものである。成果物はPLAN.md（企画書）・SPEC.md（仕様書）・TODO.md（実装計画）・KNOWLEDGE.md（検証・運用ガイド）の4ドキュメント統合体で、MVP（1戦略・1通貨ペア・12ヶ月バックテスト）から V1（複数ペア・複数フィルター）へのスケールアップを見据えている。開発期間はMVP 8週間（2026-03-06～2026-04-30）、V1完成目標は12週間（2026-06-30）である。

## 2. 市場背景＆制約条件

### 2.1 国内FX市場の現況

| 項目 | 現況 | データソース |
|------|------|------------|
| **MT4支配的地位** | 国内FXにおけるEA取引はMT4が圧倒的主流。MT5対応ブローカーは4社のみ（Philip Securities、OANDA Securities、Gaitame Finest、AvatTrade）。2026年春FXTF追加予定だが対応遅延中 | リサーチ01行目 |
| **スプレッド標準化** | USD/JPY 0.2銭以下がスタンダード。松井証券は2026年2月より段階制採用し、1万通貨以下で0.1銭に縮小 | リサーチ03行目 |
| **流動性・稼働環境** | ドル円・ユーロドル・ポンド円が標準流液性。取引量充分で24時間スプレッド・スリッページ相対安定 | リサーチ25行目 |

国内FXはMT4が実質的なEA業界標準であり、当企画はMT4を第一選択肢とする。

### 2.2 運用制約条件の確認

| 禁止事項 | 規制根拠 | 代替手法 |
|---------|---------|--------|
| **ナンピン** | 国内規制＋ブローカー約款（同一商品同方向で証拠金規制4%以上） | 固定ロット + ATRベース SL設定で1回の損失を口座資金の2%に制限 |
| **マーチンゲール** | 黙認状態だが避けるべき（Fintokeiは2025年7月28日で撤廃） | 固定リスク比率法（ケリー基準の1/4 Kelly）で機械的管理 |

**ポジション上限**: 同一通貨ペア同方向で最大1ポジション。複数ペア版（V1）では各ペア独立管理。

### 2.3 現実的な期待値設定

## 3. コンセプト3案比較分析

### 3.1 比較軸定義表（8軸以上）

| 軸 | トレンドフォロー型（EMA+ADX） | レンジ型（ボリバン+RSI） | モメンタム型（ATRブレイク） |
|---|---|---|---|
| **戦略コンセプト** | EMA短期5/9日×中期20/25日クロス + ADX14期間で強度フィルター | ボリバン20期間±2σタッチ + RSI14期間30/70フィルター | ATR14期間のブレイク幅決定 + EMA20/25日で方向フィルター |
| **推奨タイムフレーム** | 1H足（推奨） | 1H～4H足 | 1H足以上（スリッページ考慮で4H推奨） |
| **期待勝率** | 40～50% | 60%以上 | 45～55%（ダマシ含む） |
| **期待Max DD** | 15～25% | 10～15% | 12～20% |
| **強み度** | 中（信頼度・サンプル数稼ぎやすい） | 高（勝率・DD最小化） | 中（初動キャッチ） |
| **弱み度** | 高（勝率低・DD大・相場適応性） | 中（TP曖昧） | 中（ダマシ多い） |
| **リスク度** | 高（バックテスト≠リアル乖離大） | 中～高（強トレンド時に大損） | 中（ブレイク失敗頻発） |
| **開発難度** | 低（EMA・ADX標準ライブラリ豊富） | 低～中（レンジ判定ロジック複雑） | 中（ブレイク検出・EMA統合要） |

### 3.2 コンセプト詳細：案1トレンドフォロー型（EMA+ADXクロス）

**パラメータ基準**:
- EMA短期5/9日 × 中期20/25日クロス（1H足推奨）
- ADX 14期間・20以上でエントリー許可（トレンド形成段階以上）
- SL：ATR×3倍（1%前後のリスク、中期足基準）
- ロット：固定リスク率2%（リスク許容額÷SL pips÷1pip当たり価値）

**強み** (3点):
1. 高信頼度パラメータ：EMA/ADXは業界標準で最適化リスク低い
2. スプレッド耐性：トレンド保有期間長い→スプレッド0.2銭での損益影響最小
3. サンプルサイズ稼ぎやすい：1H足・複数ペア対応で2000回取引達成が容易

**弱み** (3点):
1. 勝率低：40～50%が現実的→マイナス連敗でロジック疑い強い
2. Max DD大：15～25%が現実的→20%制約で余裕わずか
3. トレンドレス相場で機能停止：ADXフィルター機能してもダマシ多数

**リスク** (3点):
1. バックテスト≠リアル乖離：スリッページ実際+3～10pips→勝率が実績値より3～5%低下可能性高い
2. スプレッド変動時の制約超過：0.2銭基準で計算したSLが拡大時に20% MaxDD超過リスク
3. 複数パラメータ過度な最適化：EMA周期・ADX強度・TP倍数の過度な調整でフォワード崩壊80%超

**開発難度**: 低

---

### 3.3 コンセプト詳細：案2レンジ型（ボリンジャーバンド+RSI逆張り）

**パラメータ基準**:
- ボリンジャーバンド 20期間・±2σ（デフォルト）
- RSI 14期間・フィルター閾値30/70
- SL：±2σ外側5～10pips
- ロット：固定リスク率2%

**強み** (3点):
1. 高勝率：60%以上が現実的→マイナス連敗が少ない→ロジック疑い軽減
2. Max DD最小：レンジ相場得意で反発待つため短期保有・リスクリワード小→DD 10～15%で済む
3. 取引機会多い：レンジ相場時間が全相場の40～60%占有→MVP 2000回達成が6～8ヶ月で可能

**弱み** (3点):
1. トレンド相場で大損：ブレイク時にボリバン上下線突き抜け→SL狩られ連敗（ダマシ多数）
2. リスクリワード悪化：反発トレードで利幅小・損失大になる局面多い
3. TP設定の恣意性：ボリバン内での目標位置が明確でない→パラメータ最適化陥穽高い

**リスク** (3点):
1. RSIダマシ：強いトレンド中もRSI70以上で「売られすぎ」と誤判定→逆張りで焼ける
2. スプレッド拡大でSL発動多発：変動スプレッド時にボリバン内での小さなSLが頻繁に刺さる
3. RSI・BB周期の過度な最適化でフォワード崩壊

**開発難度**: 低～中

---

### 3.4 コンセプト詳細：案3モメンタム型（ATRブレイク+EMA方向性フィルター）

**パラメータ基準**:
- ATR 14期間・ブレイク幅決定用
- ブレイク参照足：20足～50足直近高値/安値
- EMA 中期20/25日で上向き/下向き確認
- ロット：固定リスク率2%（ATR×2倍をSL基本）

**強み** (3点):
1. トレンド初動キャッチ：レンジ離脱時の加速局面を捉える→利益伸ばしやすい
2. ブレイク明確：エントリー/SLロジック単純・パラメータ少ない→過剰最適化リスク低い
3. ボラティリティ適応：ATRで相場状況に応じたSL自動調整→固定値より柔軟

**弱み** (3点):
1. ダマシ多い：ブレイク失敗（戻し）が30～40%確度→連敗リスク高い
2. 方向性フィルター不足：EMA単一指標では弱いトレンド区別が甘い
3. M足相場での機能不全：下位足ブレイクが上位足で抵抗になる場合→シグナル無効化

**リスク** (3点):
1. スリッページによるブレイク無効化：ブレイク確定がスリッページで吸収される可能性高い
2. TP設定の不確実性：ブレイク後の利益伸ばし方が曖昧→パラメータ多重化陥穽
3. 短期足への過度な依存：1H足以下での実装だとスプレッド・スリッページ影響極大

**開発難度**: 中

## 4. 推奨案の採択根拠

**推奨案**: **コンセプト案2：レンジ型（ボリンジャーバンド+RSI逆張り）**

### 採択理由（数値根拠付き）

#### 理由1：MaxDD制約への完全適合性

期待Max DD 10～15%で20%制約を余裕でクリア（最大余裕5～10pp）。他案との比較：
- トレンド型：15～25%（上限ギリギリ、リスク大）
- モメンタム型：12～20%（グレーゾーン）
- **レンジ型：10～15%（最安全）**

スプレッド拡大時（0.2銭→0.5銭）の想定ストレスでも、レンジ型は15%+5pp余裕を保有できる。

#### 理由2：心理的安定性と継続可能性

高勝率60%以上により、マイナス連敗が統計的に少ない（期待最大連敗8回以下）。

対比：
- トレンド型：40～50%勝率 → 期待最大連敗12～15回 → 心理的負荷極大
- **レンジ型：60%以上 → 期待最大連敗5～8回 → 心理的耐性向上**

マイナス連敗での「ロジック疑い」による感情的破綻を最小化する。

#### 理由3：MVP検証期間の短縮

取引機会が多い（レンジ相場時間が全相場の40～60%占有）。
- レンジ型：1年間で2000回取引達成が6～8ヶ月で可能 → MVP検証期間短縮
- トレンド型：同じ2000回達成に12ヶ月要する

早期GO/NO-GO判定によるV1への進捗加速が可能。

#### 理由4：実装難度と過剰最適化リスク低減

パラメータが業界標準で明確（ボリバン20期間・±2σ、RSI14期間・30/70、ATR14期間）。
- 標準設定で基本的に動作
- パラメータ調整の自由度を制限 → 過剰最適化陥穽を構造的に回避

複数パラメータ（EMA周期・ADX強度・TP倍数）を多重調整する必要がない。

## 6. リスク前提・制約事項

### 6.1 バックテスト≠リアル乖離の前提

| 乖離原因 | 構造 | 期待値低下率 |
|---------|------|----------|
| **スプレッド変動** | 固定0.2銭→変動0.3～0.5銭 | 月利0.5～1.0pp低下 |
| **スリッページ** | 実際+3～10pips（予定価格より悪化） | 月利0.5～1.5pp低下 |
| **リクオート** | ブローカー側による約定拒否・再提示 | 取引機会喪失1～3% |
| **時刻ずれ** | NTP遅延・ブローカーサーバータイムラグ | シグナル遅延300～500ms |

**複合影響**: リアル稼働では-20～40%成績悪化が典型（リサーチ46行目）。**対応戦略**：バックテスト期待値の-35%を基準に月次監視。

### 6.2 過剰最適化（聖杯EA）の危険シグナル

以下の複合該当時 → フォワード崩壊80%超の可能性（リサーチ19行目）→ **NO-GO判定**:

| # | 危険シグナル | 閾値 | 判定ロジック |
|---|-----------|------|-----------|
| 1 | **PF（過剰値）** | PF ≥ 3.0 | 利益構造が異常。スプレッド最適化の可能性 |
| 2 | **勝率（過剰高）** | 勝率 ≥ 85% | 逆張り過剰最適化またはナンピン違法取引疑い |
| 3 | **サンプル不足** | 取引数 < 500 | 統計的信頼度ゼロ。必ずNO-GO |
| 4 | **連敗数（過剰少）** | 最大連敗 ≤ 2 | 環境特異化。特定期間への過度な最適化 |

**検証対策**: ウォークフォワード分析（IS/OOS 70:30分割）での二重検証を必須実施。

### 6.3 市場環境別の制約と対応

#### 強トレンド相場での制約

レンジ型戦略の根本的弱点：トレンド相場での大損リスク。

**対応方針（V1）**: ADXフィルター追加
- ADX < 20：レンジ相場判定 → エントリー100%許可
- ADX 20～40：トレンド形成段階 → エントリー50%制限（ロット半減）
- ADX ≥ 40：強トレンド判定 → エントリー0%禁止（全停止）

#### ボラティリティ急変時の制約

コロナショック時（VIX 60+）のような異常相場では、固定SL（ATR×2）が不十分。

**対応方針（V1）**: 動的SL/TP調整ロジック搭載
- SL_pips = ATR × (基本倍数 × ボラティリティスコア)
- ボラティリティスコア = 直近ATR ÷ 20日平均ATR
- 相場静寂時：SL縮小 / ボラティリティ上昇時：SL拡大

#### スプレッド拡大時の制約

実際スプレッド（0.3銭から0.5銭へ拡大時）でのSL計算誤差。

**対応方針**: バックテスト時にスプレッド0.5銭で再検証。成績-20%以上の低下時は要確認。

## 5. ロット計算式と資金管理理論

### 5.1 固定リスク比率法

**ロット計算式**（MVP採用：固定リスク2%方式）

```
Lot = (口座残高 × リスク率%) ÷ (SL_pips × 1pip当たりの価値)

【具体例1】USD/JPY の場合（最重要）
  口座残高: 100万円
  リスク率: 2% → リスク許容額 = 100万 × 0.02 = 2万円

  USD/JPY仕様:
    1 Lot = 10万通貨
    1 pips = 1000円

  シナリオ：SL = 20pips と設定した場合
    損失額 = 20 pips × 1000円 = 2万円（リスク2%）
    Lot = 2万円 ÷ (20 × 1000) = 1.0 Lot
    → ポジションサイズ：10万通貨

【具体例2】複数取引での月利5%検証
  勝率60% / 平均利益100pips / 平均損失20pips の場合：
    利益トレード18回 × 100pips × 1000円 × 1.0Lot = 180万円
    損失トレード12回 × 20pips × 1000円 × 1.0Lot = 24万円
    純利益 = 156万円 → 初月100万円に対して月利156%

  実際の月利5%実現（ドローダウン・複数ペア分散考慮）：
    → 月間20～25回の取引と-35%乖離を見込むと月利5%が妥当
```

### 5.2 ケリー基準との関係（数学的根拠）

```
f* = (W × p - L × (1 - p)) ÷ W

where:
  f* = 最適投資比率（ケリー理論値）
  W = 平均利益（pips）
  L = 平均損失（pips）
  p = 勝率

計算例：勝率 60% / 平均利益 100pips / 平均損失 20pips
f* = (100 × 0.60 - 20 × 0.40) ÷ 100 = 0.52
→ 口座資金の52%をリスク配分すべき（理論値・危険）

実践対応（リサーチ16行目）：
  1/4 Kelly法：0.52 × 0.25 = 0.13（13%）
  MVP 固定2%方式：さらに保守的（0.02 = 2%）
  → フォワード崩壊リスク最小化
```

### 5.3 複利効果・複利計算実例

月利5%達成時の複利曲線：

| 月 | 口座残高 | リスク2% | SL20pips時Lot | 利益 | 複利後残高 |
|---|---------|---------|-------------|------|---------|
| 初月 | 100.0万 | 2.0万 | 1.00 | 5.0万 | 105.0万 |
| 2月 | 105.0万 | 2.1万 | 1.05 | 5.25万 | 110.25万 |
| 3月 | 110.25万 | 2.21万 | 1.10 | 5.51万 | 115.76万 |
| 6月 | 131.68万 | 2.63万 | 1.32 | 6.58万 | 138.28万 |
| 12月 | 179.08万 | 3.58万 | 1.79 | 8.95万 | **188.03万** |

**複利効果**：12ヶ月で100万円 → 188万円（88%増）。固定ロットでは60%増に留まる。

## 8. SL・TP設計の数学的根拠

### 8.1 SL設定：ATR × 2倍

**ATR（Average True Range）の特性**:
- 直近14足のボラティリティを数値化
- 相場静寂時→ATR小→SL狭い / 相場荒れ→ATR大→SL広い
- **適応性**：市場環境に自動調応

**ATR × 2倍 根拠**（リサーチ14行目）:
- **中期足での適正リスク**：1%前後
- **実績統計**：
  - トレンド型 15～25% MaxDD
  - レンジ型 10～15% MaxDD
- **倍数バランス**：
  - ATR × 1倍：SL狩られ多発
  - ATR × 2倍：**適度（採用）**
  - ATR × 3倍：リスク超過

**計算例（ドル円）**:
```
ATR = 20 pips → SL = 40 pips
口座100万、Lot=1.0時の損失 = 4万円（リスク4%）
→ リスク2%目標で Lot = 0.5 に自動調整

相場静寂時（ATR 10）：SL 20 pips → Lot 1.0（大）
相場荒れ時（ATR 30）：SL 60 pips → Lot 0.33（小）
→ ボラティリティに応じた自動調整
```

### 8.2 TP設定：20MA（ボリバン中心線）

**設計ポイント**:
1. **レンジ戦略特性**：反発トレードで中心線への回帰
2. **簡潔性**：指標1つで検証容易
3. **過剰最適化回避**：TP値複数試行なし

**代替案との比較**:

| TP設定案 | 利幅期待値 | 短所 | 採用判定 |
|---------|---------|------|--------|
| **20MA** | 20～40pips | やや小さい可能性 | **MVP推奨** |
| BB_Middle±5 | 20～35pips | 追加パラメータ | 非推奨 |
| 固定50pips | 全期間同じ | 環境非適応 | 非推奨 |
| RSI中点(50) | 20～50pips | ロジック複雑 | 非推奨 |

**採用理由**：MVP検証優先に複雑さ排除。業界標準ボリバン中心線を信頼。

| 2 | RSI計算 | 14期間での値 | RSI値 0～100範囲内 | 範囲内 + 精度小数点2位 |
| 3 | ATR計算 | 14期間でのボラティリティ | ATR>0 / 前足より過度に増減しない | 物理的妥当性確認 |
| 4 | ロット計算 | 口座100万・SL20pips→Lot計算 | Lot=0.5を確認 | 誤差なし |
| 5 | エントリー判定 | BB接触+RSI30→売りシグナル | True を返す | 判定ロジック正確性 |
| 6 | SL/TP計算 | ATR=20→SL=40・TP=20MA値 | 妥当範囲内 | 計算根拠との整合性 |

**注記**：単体テストは各モジュール開発直後に実施。計算精度が全体品質を左右するため、特に Module 1・3 は念入り。

## 4. MQL4/MQL5 既知の罠・バグ予防策

### 4.1 計算精度・インデックスエラーの罠

**罠1：iBarShift・iClose の履歴データ不足**

```
症状：
  バックテスト途中から勝率が0%になる。計算値が NaN または -1 を返す

原因：
  iBarShift(symbol, timeframe, datetime, exact) を使用時、
  過去足のデータが不足していると -1（見つからない）を返す

対策：
  [ ] iBarShift 戻り値を必ずチェック（-1 判定）
  [ ] エラー発生時は OnInit() でスキップフラグを立てる
  [ ] 代わりに「足 shift（現在から N足前）」の固定インデックスを使用推奨
  例：int pos = iBarShift(Symbol(), Period(), TimeCurrent(), false);
      if(pos > 0) { Close[pos] を使用 } else { Skip }
```

**罠2：配列インデックスとバー番号の逆転**

```
症状：
  MT4 では バー [0] = 最新足、バー [1] = 1足前
  MT5 では 逆順（ArraySetAsSeries() で整合）

原因：
  MT4/MT5 混在コーディングで逆順混同

対策：
  [ ] MT4 コードは必ずコメント：// Bar[0]=Latest
  [ ] MT5 では最初に ArraySetAsSeries(Close, false) を実行
  [ ] テスト時に「直近10足の Close[0]～Close[9]」を出力確認
```

**罠3：iOpen/iHigh/iLow/iClose の時刻ずれ**

```
症状：
  バックテストと リアルタイム足の値が異なる（足が確定していない）

原因：
  OnBarOpen イベント中に未確定の足データを参照

対策：
  [ ] OnTick() ではなく、足確定時のタイミングで判定
  [ ] Close[1]（前足確定値）を使用。Close[0]（現在足未確定値）は避ける
  例：if(Bars > 10 && iBarShift(Symbol(), 0, iTime(Symbol(), PERIOD_H1, 0)) == 1)
      { /* 足が新規に確定 */ }
```

### 4.2 ポジション・オーダー管理の罠

**罠4：OrderSelect の戻り値未チェック（致命的バグ）**

```
症状：
  ポジションが正しくクローズされない。不意のダブルクローズ試行。

原因：
  OrderSelect(ticket, SELECT_BY_TICKET, MODE_TRADES) の戻り値をチェック忘れ
  False（失敗）でも処理を続行してしまう

対策：
  [ ] 必ず以下のように書く：
      if(OrderSelect(ticket, SELECT_BY_TICKET, MODE_TRADES)) {
        if(OrderClose(...)) { /* 成功 */ }
        else { /* エラーハンドリング */ }
      } else {
        Print("OrderSelect Failed: ", GetLastError());
      }
  [ ] GetLastError() でエラーコード確認（4xxx = 無効なチケット等）
```

**罠5：同一足での複数エントリー**

```
症状：
  1本の足で複数回エントリーが発火し、同方向ポジション複数作成

原因：
  OnTick() 内でエントリー条件判定 → 同一足の複数 OnTick 呼び出しで条件成立

対策：
  [ ] エントリー条件の外側に「新規足確定」フラグを設定
  [ ] 静的変数 static bool NewBar = false を使用
  例：
      if(iBarShift(Symbol(), 0, iTime(Symbol(), PERIOD_H1, 0)) > 0) {
        NewBar = true;
      } else {
        NewBar = false;
      }
      if(NewBar && (ボリバン+RSI条件)) { OrderSend() }
```

**罠6：MaxDD 計算での未決済ポジション無視**

```
症状：
  MaxDD がバックテストでは小さいが、リアルでは大きくなる

原因：
  バックテスト終了時に未決済ポジションがある場合、
  MaxDD 計算から漏れている（ターミナルログを見ると分かる）

対策：
  [ ] OnDeinit() で全ポジションをクローズする強制処理を入れる
  [ ] バックテスト結果で「未決済ポジション 0個」であることを確認
  [ ] ウォークフォワード分析でも同じチェック
```

### 4.3 リペイント・シグナル遅延の罠

**罠7：前足データ参照による誤シグナル（リペイント）**

```
症状：
  バックテストでは勝率70%だが、フォワード・リアルでは50%に低下

原因：
  指標計算で Close[0]（未確定の現在足）を参照 → 足が確定したら値が変わる（リペイント）
  → バックテストでは「未来情報を知った」状態で最適化されている

対策：
  [ ] Close[1] 以前（確定済み足）のみを指標計算に使用
  [ ] ボリバン・RSI・ATR全て「期間14」→ Close[1]～Close[14] の範囲を参照
  [ ] Close[0]（現在足未確定値）はあくまで「参考値」に留める
  例：
      double bb_upper = iBands(Symbol(), PERIOD_H1, 20, 2, 0, PRICE_CLOSE, MODE_UPPER, 1);
      // 1 = 前足（確定）を参照。0 だと現在足（未確定）を参照 → リペイント
```

**罠8：インジケーターの計算ラグ（遅延）**

```
症状：
  エントリーシグナルが実際のターニングポイントより遅い（既にトレンド反転後）

原因：
  移動平均線・ボリバンは過去14～20足のデータを使用
  → 必然的に価格動きより 7～10足遅れる

対策：
  [ ] 遅延を許容できない戦略の場合：短期パラメータ（7日線等）を使用
  [ ] ただしパラメータを安易に変更すると過度な最適化リスク
  [ ] MVP では「遅延＝安全性」と見なし、標準パラメータを守る
```

## 6. パラメータドリフト検出マニュアル（V1向け）

### 6.1 パラメータドリフトとは

**定義**

```
パラメータドリフト：
バックテスト期間内で最適化されたパラメータが、
新しい相場環境では機能しなくなる現象

例：
- 2024年のトレンド相場で最適化された EMA パラメータが、
  2025年のレンジ相場では勝率30%に低下
```

**ドリフト検出法：ウォークフォワード分析（IS/OOS）**

```
全期間：2024-01-01 ～ 2026-03-06（27ヶ月）

分割1：
  IS（学習）: 2024-01 ～ 2024-11（11ヶ月）→ 最適パラメータ算出
  OOS（検証）: 2024-12 ～ 2025-02（3ヶ月）→ 新パラメータの成績測定

分割2（ローリング）:
  IS: 2024-03 ～ 2025-01（11ヶ月）→ 再最適化
  OOS: 2025-02 ～ 2025-04（3ヶ月）→ 測定

分割3（同様ローリング）:
  IS: 2024-05 ～ 2025-03（11ヶ月）
  OOS: 2025-04 ～ 2025-06（3ヶ月）

ドリフト判定：
  乖離 ≤ -20% → 正常範囲（想定内）
  乖離 -20% ～ -40% → 注意（パラメータ調整検討）
  乖離 > -40% → 警告（ロジック見直し必須）
```

### 6.2 V1以降での月次パラメータドリフト検出手順

**自動検証フレームワーク実装案**

```
V1 実装予定：月次自動ウォークフォワード検証

実行スケジュール（毎月末）:
1. 過去30ヶ月のデータセット準備
2. 10週間単位で分割（3ローリング = 30週間カバー）
3. 各分割で IS/OOS 70:30 テスト実行
4. OOS 成績の平均を計算
5. IS との乖離率を自動計算
6. 乖離 > -40% ならアラート自動生成

検出後の対応フロー：
ドリフト検出
    ↓
【判定】OOS vs IS 乖離率
    ├─ ≤ -20% → 継続稼働（次月監視）
    ├─ -20% ～ -40% → パラメータ微調整検討
    │  └─ 新パラメータで再ウォークフォワード実施
    └─ > -40% → 【停止】ロジック見直し必須
         └─ 根本的な戦略変更 or 新コンセプト検討
```
