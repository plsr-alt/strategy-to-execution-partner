### 4. 推奨案の採択根拠

#### 執筆論点

**推奨案**: **コンセプト案2：レンジ型（ボリンジャーバンド+RSI逆張り）**

##### 採択理由（段落3～4個）

1. **MaxDD制約への適合性**
   - 期待Max DD 10～15% で20%制約を余裕でクリア（最大余裕5～10pp）
   - 他案（トレンド15～25%、モメンタム12～20%）との比較で最も安全

2. **心理的安定性と継続可能性**
   - 高勝率60%以上 → マイナス連敗が少ない → ロジック疑い軽減
   - トレンド型（40～50%勝率）の心理的負荷が大きい

3. **MVP検証期間の短縮**
   - 取引機会多い（レンジ相場時間40～60%占有）
   - 1年間で2000回取引達成が6～8ヶ月で可能
   - 他案では12ヶ月要する

4. **実装難度と過剰最適化リスク**
   - パラメータが明確（ボリバン・RSI デフォルト + SL明確）
   - 業界標準設定で最適化陥穽少ない

**執筆粒度**: 表1行（原因構造）+ 説明1段落

- **聖杯EA判定基準**: PF ≥3.0 OR 勝率≥85% OR サンプル<500
  - 上記複合該当時 → フォワード崩壊80%超の可能性

### 5. ロット計算式と資金管理理論

#### 執筆論点

##### 5.1 固定リスク比率法（ケリー基準・固定比率ポジションサイジング）

**ロット計算式**（推奨：固定リスク2%方式）

```
Lot = (口座残高 × リスク率%) ÷ (SL_pips × 1pip当たりの価値)

例：
  口座残高: 100万円
  リスク率: 2%
  → リスク許容額 = 100万 × 0.02 = 2万円

  USD/JPY の場合：
  1 Lot = 10万通貨
  1 pips = 1000円

  SL = 20pips と設定した場合
  → 損失額 = 20 pips × 1000円 = 2万円（リスク2%）

  Lot = 2万円 ÷ (20 pips × 1000円) = 1 Lot
```

**執筆粒度**: 式 + 具体例3～4個（USD/JPY・EUR/USD・GBP/JPY別）

##### 5.2 ケリー基準との関係（数学的根拠）

**ケリー基準（Kelly Criterion）の理論背景**

```
f* = (W × p - L × (1 - p)) ÷ W
where:
  f* = 最適投資比率（ケリー理論値）
  W = 平均利益（pips）
  L = 平均損失（pips）
  p = 勝率
```

**実践上の注意**（16行目より）

- **理論値は危険**: ケリー基準の直接適用は過度なレバレッジを招く
- **1/4 Kelly法**: 実践ではケリー値の1/4～1/3に縮小することが推奨
- **MVPでの適用**: 固定リスク2%方式で間接的にケリーの保守的運用を実現

**執筆粒度**: 式 + 説明段落2～3個

##### 5.3 固定比率ポジションサイジングの論理

**概念図**（テキスト形式）

```
固定リスク2%方式の特性：
・利益が出 → 口座残高増 → 自動的にLot数増加
・損失が出 → 口座残高減 → 自動的にLot数減少
→ 複利効果と下落圧力が動的に自己調整
```

**複利計算例**:

### 8. SL・TP設計の数学的根拠

#### 執筆論点

##### 8.1 SL設定：ATR × 2倍 の理由

**ATR（Average True Range）とは**:
- 直近14足のボラティリティを反映する指標
- 相場が静寂な時は ATR 小 → SL 狭い
- 相場が荒れている時は ATR 大 → SL 広い
- **適応性**: 市場環境に自動調応

**ATR × 2倍 選定根拠**（14行目より）:
- **中期足での適正リスク**: 1%前後（口座資金当たり）
- **実績統計**: トレンド型15～25%, レンジ型10～15% のMaxDD経験値
- **余裕度**: 1倍だと狭すぎてSL狩られ多発、3倍だと広すぎてリスク超過

**計算例**:

```
ATR = 20 pips （その日の市場ボラティリティ）
SL = ATR × 2 = 40 pips
口座100万、SL40pips、1Lot時損失 = 40pips × 1000円 = 4万円（4%）
リスク率2%目標を達成するには Lot = 0.5（50,000通貨）に抑制
```

**執筆粒度**: 説明段落2～3個 + 計算例1個

##### 8.2 TP設定：20MA（ボリバン中心線） の理由

**TP目標の選定ポイント**:
- **レンジ戦略の特性**: 反発トレードなので、バンドの中心線（20MA）への回帰を利用
- **簡潔性**: 指標が1つ（ボリバン中心 = 20MA）で検証が容易
- **過度な最適化回避**: TP値を複数試行しない → パラメータドリフト低い

**代替案との比較**:

| TP設定案 | 利幅期待値 | 短所 |
|---------|---------|------|
| **20MA（採用案）** | 20～40pips（相場依存） | やや小さい可能性 |
| BB_Middle ± 5pips | 同上 | ほぼ同値（追加パラメータ） |
| 固定50pips | 全期間同じ | 相場環境非適応 |
| RSI中点（50）到達時 | 20～50pips（広い） | 判定が複雑 |

**採用理由**: MVPでの検証を最優先に、複雑さ排除を重視

**執筆粒度**: 説明段落1～2個 + 比較表1個

**責務説明（短編・1モジュール1段落）**

##### Module 1：インジケーター計算エンジン
ボリンジャーバンド・RSI・ATR・移動平均の4つの指標値を足ごとに計算し、次のロジック層に供給する。計算のバグ（過去データ参照ミス・インデックスずれ）が全体の精度に直結するため、単体テスト必須。

##### Module 2：エントリー判定ロジック
ボリバン上下限への接触判定とRSI 30/70交差判定を組み合わせて、売りエントリー・買いエントリーの可否を判定。「ボリバン + RSI」の複合フィルターでダマシを低減。逆に判定甘くするとレンジ相場内で無駄なエントリーが増加するため、パラメータ調整後は必ず統合テストで再検証。

##### Module 3：ポジション管理・資金管理
成立したエントリーから、SL値・TP値・ロット数を計算して確定。固定リスク2%方式によるロット計算が複利効果を生むため、ロット計算エラーは複数月で累積損失に直結。また、口座残高の増減によって Lot が動的に変化することを確認するテストケースを用意。

##### Module 4：終了条件・クローズロジック
TP・SL・強制クローズ（72時間保有）の3条件のいずれかを満たしたポジションをクローズして P&L を確定。クローズされないポジションがあると統計計算が破綻するため、バックテストログで「未決済ポジション 0個」であることを確認。

##### Module 5：ログ・統計・アラート
取引ごとの履歴（エントリー時刻・価格・SL・TP・クローズ価格・P&L）を記録し、勝率・Profit Factor・MaxDD・連敗回数を計算。MVPでは簡易版（ターミナルに出力）で十分だが、V1ではCSV化＋グラフ可視化を実装。

### 4. MQL4/MQL5 既知の罠・バグ予防策

#### 執筆論点

##### 4.1 計算精度・インデックスエラーの罠

**罠1：iBarShift・iClose の履歴データ不足**

```
症状：
  バックテスト途中から勝率が0%になる。計算値が NaN または -1 を返す

原因：
  iBarShift(symbol, timeframe, datetime, exact) を使用時、
  過去足のデータが不足していると -1（見つからない）を返す

対策：
  [ ] iBarShift 戻り値を必ずチェック（-1 判定）
  [ ] エラー発生時は OnInit() でスキップフラグを立てる
  [ ] 代わりに「足 shift（現在から N足前）」の固定インデックスを使用推奨
  例：int pos = iBarShift(Symbol(), Period(), TimeCurrent(), false);
      if(pos > 0) { Close[pos] を使用 } else { Skip }
```

**罠2：配列インデックスとバー番号の逆転**

```
症状：
  MT4 では バー [0] = 最新足、バー [1] = 1足前
  MT5 では 逆順（ArraySetAsSeries() で整合）

原因：
  MT4/MT5 混在コーディングで逆順混同

対策：
  [ ] MT4 コードは必ずコメント：// Bar[0]=Latest
  [ ] MT5 では最初に ArraySetAsSeries(Close, false) を実行
  [ ] テスト時に「直近10足の Close[0]～Close[9]」を出力確認
```

**罠3：iOpen/iHigh/iLow/iClose の時刻ずれ**

```
症状：
  バックテストと リアルタイム足の値が異なる（足が確定していない）

原因：
  OnBarOpen イベント中に未確定の足データを参照

対策：
  [ ] OnTick() ではなく、足確定時（bar shift==0 から bar shift==1）のタイミングで判定
  [ ] Close[1]（前足確定値）を使用し、Close[0]（現在足未確定値）は避ける
  例：if(Bars > 10 && iBarShift(Symbol(), 0, iTime(Symbol(), PERIOD_H1, 0)) == 1)
      { /* 足が新規に確定 */ }
```

**执笔粒度**: 各罠3～4行説明 + 対策チェックボックス

##### 4.3 リペイント・シグナル遅延の罠

**罠7：前足データ参照による誤シグナル（リペイント）**

```
症状：
  バックテストでは勝率70%だが、フォワード・リアルでは50%に低下

原因：
  指標計算で Close[0]（未確定の現在足）を参照 → 足が確定したら値が変わる（リペイント）
  → バックテストでは「未来情報を知った」状態で最適化されている

対策：
  [ ] Close[1] 以前（確定済み足）のみを指標計算に使用
  [ ] ボリバン・RSI・ATR全て「期間14」→ Close[1]～Close[14] の範囲を参照
  [ ] Close[0]（現在足未確定値）はあくまで「参考値」に留める
  例：
      double bb_upper = iBands(Symbol(), PERIOD_H1, 20, 2, 0, PRICE_CLOSE, MODE_UPPER, 1);
      // 1 = 前足（確定）を参照。0 だと現在足（未確定）を参照 → リペイント
```

**罠8：インジケーターの計算ラグ（遅延）**

原因：
  移動平均線・ボリバンは過去14～20足のデータを使用
  → 必然的に価格動きより 7～10足遅れる

対策：
  [ ] 遅延を許容できない戦略の場合：短期パラメータ（7日線等）を使用
  [ ] ただしパラメータを安易に変更すると過度な最適化リスク
  [ ] MVP では「遅延＝安全性」と見なし、標準パラメータを守る

### 6. 後続 V1・V2 に向けたパラメータドリフト検出マニュアル

#### 執筆論点

##### 6.1 パラメータドリフトとは何か

**定義**

```
パラメータドリフト：
バックテスト期間内で最適化されたパラメータが、
新しい相場環境では機能しなくなる現象

例：
- 2024年のトレンド相場で最適化された EMA パラメータが、
  2025年のレンジ相場では勝率30%に低下
```

**ドリフト検出法：ウォークフォワード分析（IS/OOS）**

```
全期間：2024-01-01 ～ 2026-03-06（27ヶ月）

分割1：
  IS（学習）: 2024-01 ～ 2024-11（11ヶ月）→ 最適パラメータ算出
  OOS（検証）: 2024-12 ～ 2025-02（3ヶ月）→ 新パラメータの成績測定

分割2（ローリング）:
  IS: 2024-03 ～ 2025-01（11ヶ月）→ 再最適化
  OOS: 2025-02 ～ 2025-04（3ヶ月）→ 測定

分割3（同様ローリング）:
  IS: 2024-05 ～ 2025-03（11ヶ月）
  OOS: 2025-04 ～ 2025-06（3ヶ月）
```

**ドリフト判定**

```
OOS 成績 vs IS 成績の乖離を計測：

- 乖離 ≤ -20% → 正常範囲（想定内）
- 乖離 -20% ～ -40% → 注意（パラメータ調整検討）
- 乖離 > -40% → 警告（ロジック見直し必須）
```

**執筆粒度**: 定義説明1段落 + 分割例テキスト + 判定基準表

---

##### 6.2 V1 以降での月次パラメータドリフト検出手順

**自動検証フレームワーク実装案**

```
V1 実装予定：月次自動ウォークフォワード検証

実行スケジュール（毎月末）:
1. 過去30ヶ月のデータセット準備
2. 10週間単位で分割（3ローリング = 30週間カバー）
3. 各分割で IS/OOS 70:30 テスト実行
4. OOS 成績の平均を計算
5. IS との乖離率を自動計算
6. 乖離 > -40% ならアラート自動生成
```

**検出後の対応フロー**

```
ドリフト検出
    ↓
【判定】OOS vs IS 乖離率
    ├─ ≤ -20% → 継続稼働（次月監視）
    ├─ -20% ～ -40% → パラメータ微調整検討
    │  └─ 新パラメータで再ウォークフォワード実施
    └─ > -40% → 【停止】ロジック見直し必須
         └─ 根本的な戦略変更 or 新コンセプト検討
```

**執筆粒度**: フロー図 + 説明段落2～3個
